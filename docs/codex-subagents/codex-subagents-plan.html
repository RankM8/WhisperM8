WhisperM8 · Konzept + Implementierungsplan

01 · Konzept

Drei Codex-Welten, eine Session-Substanz.

Ein Codex-Subagent ist ein headless codex exec-Lauf mit Auftrag, Working Directory, Sandbox-Level und Lebenszyklus. Er ist kein Ersatz für interaktive Codex-Chats — aber weil codex exec seine Sessions als ganz normale Rollout-JSONLs in ~/.codex/sessions/ persistiert, sind alle drei Welten unten dieselbe Session-Art. Das ist der Trick, der die Übernahme in einen interaktiven Chat trivial macht.

Bestand · bleibt unverändert

Interaktive Chats

PTY + SwiftTerm-TUI, wie heute. Vom User gestartet und bedient.

  • AgentCommandBuilder baut argv (new / resume)
  • Runtime-Watcher liest Status aus Rollout-JSONL
  • Approval-Modi interaktiv, wie vom User konfiguriert
Neu · dieses Projekt

Subagent-Jobs

Headless, vom whisperm8-CLI superviselt. Gestartet von Claude Code, dir oder später Jarvis.

  • codex exec --json, ein Turn pro Aufruf
  • Folge-Turns via codex exec resume
  • Report-Vertrag per --output-schema
  • Zustand im Job-Verzeichnis, App beobachtet nur
Später · konsumiert die Jobs

02 · Architektur & Datenfluss

Das CLI superviselt, die App beobachtet.

Kernentscheidung: Variante B — das whisperm8-Binary forkt sich selbst in einen detachten Supervisor-Modus (das CLIModeDetector-Muster gibt das her). Kein IPC zur App nötig, Subagents laufen auch ohne laufende App. Die App konsumiert das Job-Verzeichnis über FSEvents — exakt die Rollenverteilung, die sie heute schon gegenüber ~/.claude/jobs/ einnimmt.

Auslöser

Claude Code · User · Jarvis

whisperm8 agent run --wait --json --parent $CLAUDE_SESSION_ID "Implementiere X, teste, committe bei grün"

argv → CLIModeDetector erkennt „agent" → CLI-Modus
whisperm8 CLI · Frontend-Prozess

agent run — Job anlegen & Supervisor forken

Short-ID generieren, Job-Verzeichnis + state.json (spawning) schreiben, detachten Supervisor-Prozess starten (whisperm8 agent-supervise <id>, setsid). Bei --wait: auf state.json-Übergang warten, Report auf stdout. Sonst: Short-ID ausgeben, exit 0.

posix_spawn · LoginShellEnvironment · commandPath("codex")
whisperm8 CLI · Supervisor-Prozess (detacht)

agent-supervise — besitzt den Codex-Prozess

Startet codex exec --json --sandbox workspace-write --ask-for-approval never --cd <repo> --output-schema report.json - (Prompt via stdin). Streamt JSONL-Events nach events.jsonl, parst thread.started → codexThreadID, schreibt State-Übergänge + PID in state.json, extrahiert finale Message nach last-message.txt.

JSONL-Events · turn.completed / turn.failed
Kontrakt auf Disk

Job-Verzeichnis ~/Library/Application Support/WhisperM8/agent-jobs/<short-id>/

state.json (State-Machine + Metadaten) · events.jsonl (roher Event-Stream, append-only) · last-message.txt (Report des letzten Turns). Spiegelt bewusst die ~/.claude/jobs/-Welt von SupervisorJobReader.

FSEvents auf agent-jobs/ (Muster: AgentDirectoryEventMonitor)
WhisperM8 App · reiner Beobachter

Sidebar · Subagent-Detail-View · Übernahme

AgentJobDirectoryMonitor entdeckt Jobs, Sidebar zeigt sie unter Parent-Session, Detail-View rendert Transcript live, Composer sendet Folge-Turns über agent send. „Interaktiv übernehmen" öffnet PTY-Tab via codex resume <thread-id>.

--wait / Background-Bash-Task endet → Harness re-invoked Claude
Rückkanal

Strukturierter Report an Claude Code

Schema-validiertes JSON: {status, summary, filesChanged, commits, testsRun, openQuestions}. Claude wertet aus, reviewt ggf. den Branch, steuert per agent send nach.

Querschnitt

~/.codex/sessions/ als gemeinsames Substrat

Jeder Subagent-Turn landet zusätzlich als reguläres Rollout-JSONL in Codex' eigenem Session-Store. Dadurch greift die bestehende Maschinerie ohne Änderung:

  • CodexTranscriptReader parst das Transcript für die Detail-View
  • AgentSessionIndexer findet die Session beim Scan
  • AgentSessionRuntimeWatcher kann das Rollout-File beobachten
  • codex resume <id> öffnet sie interaktiv — der Übernahme-Pfad

Konsequenz: kein --ephemeral verwenden — die Persistenz ist Feature, nicht Nebeneffekt.

Warum nicht Variante A / C?

Verworfene Alternativen

  • A — App spawnt, CLI als IPC-Client: es gibt heute bewusst kein IPC (CLI ist App-unabhängig), und Subagents müssen auch ohne laufende App funktionieren
  • C — launchd-Jobs: robust, aber Plist-Management + Cleanup sind überdimensioniert für Jobs mit Minuten-Lebensdauer
Bekannte Gotchas · gelten auch hier

Aus CLAUDE.md übernommen

  • Spawn ausschließlich über LoginShellEnvironment + AgentCommandBuilder.commandPath("codex") — GUI/CLI-PATH ist minimal
  • NO_COLOR=1 / CLICOLOR=0 setzen (Muster: BackgroundAgentSpawner)
  • Neue Subcommands in den CLIModeDetector eintragen (argv[1]-Erkennung), sonst startet die GUI
  • Store-Mutation-Closures dürfen keine Subprozesse ausführen — Job-Lookups vorher hoisten
03 · CLI-Spezifikation

Die Oberfläche, die Claude Code bedient.

Provider-agnostisch benannt (agent, nicht codex) — v1 kennt nur Codex, aber --provider claude kann später an claude --bg durchreichen, und Claude Code hätte eine einheitliche Verwaltungsschicht für beide. Jeder Befehl hat --json für maschinenlesbare Ausgabe; Exit-Codes sind Teil des Vertrags.

agent run [flags] "<prompt>"
Job starten. Ohne --wait: gibt sofort {shortId} aus (async). Mit --wait: blockiert bis Turn-Ende, gibt vollen Report aus. Flags: --cd <dir> (Default: cwd), --model, --effort, --sandbox read-only|workspace-write (Default: workspace-write), --worktree (Opt-in: Job in frischem Git-Worktree), --allow-network (Opt-in: erlaubt u.a. git push), --parent <session-id> (Verknüpfung zur spawnnenden Claude-Session), --json.
agent send <short-id> [--wait] "<prompt>"
Folge-Turn. Intern codex exec resume <codexThreadID>, wieder über den Supervisor. Verweigert mit Exit 3, wenn Job running (Turn läuft schon) oder takenOver ist.
agent list [--json]
Alle Jobs mit shortId, State, Intent (gekürzt), cwd, Alter. Räumt dabei verwaiste Jobs auf (PID-Liveness-Check, siehe Lifecycle).
agent status <short-id> [--json]
Zustand eines Jobs — State, codexThreadID, Turn-Zähler, letzter Report (falls done). Das Polling-Primitive für Claude Code.
agent logs <short-id> [--tail N]
Zwischenstände. Menschenlesbares Rendering der letzten Events aus events.jsonl (Agent-Messages, Tool-Calls kompakt). Für „was macht der gerade?" mitten im Lauf.
agent stop <short-id>
Laufenden Turn abbrechen. SIGTERM an den Codex-Prozess (via Supervisor-PID), State → stopped. Die Codex-Session bleibt resumierbar.
agent rm <short-id>
Job entfernen. Löscht das Job-Verzeichnis (+ Worktree, falls von uns angelegt und clean — sonst Warnung + Exit 4). Die Codex-Session in ~/.codex/sessions/ bleibt unangetastet.
Ausgabe · agent run --wait --json (Exit 0 = done, 2 = failed)
{ "shortId": "a3f81c2e", "state": "done", "codexThreadID": "0197f3a2-8c41-7e02-b5d1-9a4e6c2f8b10", "turns": 1, "report": { // schema-validiert, siehe Sektion 04 "status": "success", "summary": "OutputModeStore-Migration implementiert, 14 Tests grün, committed.", "filesChanged": ["WhisperM8/Services/Dictation/OutputModeStore.swift"], "commits": [{ "sha": "9c2e1af", "message": "feat(modes): Migration v2→v3" }], "testsRun": { "command": "swift test --filter OutputMode", "passed": true }, "openQuestions": [] }, "worktree": null // oder { path, branch } bei --worktree }
Exit-Codes · Teil des Vertrags

Damit Claude ohne Text-Parsing entscheiden kann

  • 0 — ok / Job done
  • 1 — Usage-Fehler (unbekanntes Flag, fehlende ID)
  • 2 — Job failed (turn.failed, Codex-Fehler, Report sagt failure)
  • 3 — Zustandskonflikt (send auf running/takenOver)
  • 4 — Umgebungsproblem (codex-Binary fehlt, Worktree dirty, …)
Sicherheits-Defaults

Headless heißt: keine Approval-Hänger

  • --ask-for-approval never immer — ein Subagent, der auf Approval wartet, ist headless wertlos
  • --sandbox workspace-write Default — arbeiten + committen ja, Netzwerk nein
  • git push scheitert an der Sandbox — genau richtig; --allow-network setzt die Codex-Config-Option explizit
  • read-only für reine Analyse-/Review-Jobs (und Jarvis)
04 · Job-Lifecycle & Datenmodell

Eine kleine State-Machine, ehrlich auf Disk.

state.json ist die einzige Quelle der Wahrheit über einen Job. Jeder Übergang wird atomar geschrieben (temp + rename). Weil der Supervisor ein normaler Prozess ist, der sterben kann (Neustart, Crash), enthält der State die PID — und jeder Leser validiert Liveness, bevor er running glaubt.

spawning
Frontend hat das Job-Verzeichnis angelegt, Supervisor startet gerade. Kurzlebig; hängt der State hier >30s ohne lebende PID → als failed behandeln.
running
Codex-Turn läuft. Nur gültig, wenn die Supervisor-PID lebt (kill -0). Tote PID + running = orphaned → Leser korrigiert auf failed mit Grund „supervisor died".
done
Turn regulär beendet (turn.completed), Report in last-message.txt validiert. Job ist per agent send fortsetzbar → geht dann wieder auf running.
failed
turn.failed, Schema-Validierung des Reports fehlgeschlagen, oder Supervisor-Tod. failureReason im State. Auch failed-Jobs sind resumierbar (send startet neuen Turn auf der Session).
stopped
Per agent stop oder App-UI abgebrochen. Codex-Session bleibt intakt und resumierbar.
takenOver
Du hast den Job als interaktiven Chat übernommen (PTY via codex resume). Der Supervisor zieht sich dauerhaft zurück: send verweigert (Exit 3), Composer in der App deaktiviert. Exklusivität verhindert verschränkte Historien. Rückweg („wieder headless") ist bewusst v2.
state.json · Kontrakt zwischen CLI-Supervisor und App
{ "version": 1, "shortId": "a3f81c2e", "provider": "codex", "state": "running", "intent": "Implementiere X, teste, committe bei grün", "cwd": "/Users/giulianocosta/repos/whisperm8", "codexThreadID": "0197f3a2-…", // aus thread.started; null bis Turn 1 startet "parentSessionID": "c71d…", // Claude-Session, die gespawnt hat (optional) "supervisorPid": 48213, // Liveness-Anker "turns": 1, "sandbox": "workspace-write", "worktree": null, // oder { "path": "…", "branch": "subagent/a3f81c2e" } "failureReason": null, "createdAt": "2026-07-04T14:02:11Z", "updatedAt": "2026-07-04T14:06:48Z" }
Report-Schema · via codex exec --output-schema erzwungen
// Der Subagent MUSS seinen letzten Turn mit diesem JSON beenden — Codex validiert selbst. { "status": "success | partial | failure", "summary": "string — 2-5 Sätze, was getan wurde und was offen ist", "filesChanged": ["string"], "commits": [{ "sha": "string", "message": "string" }], "testsRun": { "command": "string", "passed": boolean } // oder null, "openQuestions": ["string"] }
Konvention

Warum ein eigenes Verzeichnis, nicht ~/.claude/jobs?

  • ~/.claude/ und ~/.codex/ gelten im Projekt als extern und read-only — das bleibt so
  • Unser Job-Verzeichnis lebt im eigenen App-Support-Ordner neben AgentSessions.json
  • Struktur spiegelt bewusst SupervisorJobReaders Welt — ein neuer AgentJobReader ist fast 1:1 danach modellierbar
Turn-Semantik

Ein Turn = ein codex-exec-Prozess

  • Zwischen Turns läuft kein Prozess — der Job ist „idle auf Disk", nur State + Session existieren
  • agent send startet einen frischen Supervisor für den Resume-Turn
  • Das macht Jobs robust gegen Neustarts: nichts Langlebiges muss überleben

States · Sidebar-Row

Schlankeres Zustandsmodell als Chats

  • Mint pulsierend = running (Turn läuft)
  • Blau = done mit ungelesenem Report — derselbe „schau mich an"-Moment wie bei fertigen Chats; füttert später Jarvis' Attention-Sorting
  • Rot = failed, grau = stopped/gelesen
  • Kein awaitingInput — approval never macht den Zustand unmöglich; taucht er doch auf, ist es ein Bug
Interactions

Alles über bestehende Muster

  • Composer → agent send: UI ruft dieselbe CLI-Logik auf (geteilter Code, kein Shell-out aus der App nötig)
  • „→ In Claude-Chat einfügen": Report als Prompt in die Parent-Session routen — die Diktier-Pipeline hat die Maschinerie (Prompt-Routing in aktive Chats) bereits
  • Kontextmenü: Logs, Stop, Übernehmen, Entfernen — analog Background-Agent-Menü
  • Zähler-Chip an Parent-Row („2 ⚙"): Klick expandiert die Kinder

06 · Claude-Code-Integration

Nutzungs-Parität zu nativen Subagents — und zwei Extras.

Ein nativer Subagent ist aus Nutzungssicht: Prompt rein → isolierte Arbeit → Report zurück → Parent benachrichtigt. Genau das liefert die CLI-Schicht — die Benachrichtigung erledigt Claude Codes eigenes Harness, wenn der Aufruf als Background-Bash-Task läuft. Extra 1: laufende Jobs sind per send nachsteuerbar. Extra 2: du siehst jeden Subagent-Chat vollständig in der App.

Nutzungsmuster aus Claude Codes Sicht
# Kurz (< ~2 Min erwartete Laufzeit): synchron whisperm8 agent run --wait --json --parent $CLAUDE_SESSION_ID \ "Reviewe den Diff auf main gegen HEAD~3 auf Regressionen. Nur Analyse." \ --sandbox read-only # Lang: als Background-Task starten — das Harness meldet sich, wenn der Prozess endet. # (Claude nutzt run_in_background; kein Polling-Loop nötig.) whisperm8 agent run --wait --json --worktree --parent $CLAUDE_SESSION_ID \ "Implementiere X, führe die Tests aus, committe bei grün." # Zwischenstand prüfen / nachsteuern, während der Turn läuft bzw. danach whisperm8 agent logs a3f81c2e --tail 20 whisperm8 agent send a3f81c2e --wait --json "Klärung: mappe Legacy-Modes NICHT auf den Default, wirf stattdessen."
Skill-Definition

Damit Claude das Werkzeug richtig benutzt

  • Skill oder Custom-Agent im Repo (.claude/skills/codex-subagent/ o.ä.) beschreibt die Befehle, Exit-Codes und das Report-Schema
  • Regeln darin: --parent immer setzen; ab erwarteter Laufzeit >2 Min Background-Weg statt synchronem --wait; bei schreibenden Aufgaben, während Claude selbst editiert, --worktree empfehlen (bleibt aber Opt-in)
  • Bei langlaufenden Jobs: nicht blind warten — logs --tail prüfen, ggf. send
Ehrliche Lücken vs. native Subagents

Bewusst akzeptiert bzw. adressiert

  • Getrennte Regel-Universen: Codex liest AGENTS.md + ~/.codex/config.toml, nicht CLAUDE.md. Gegenmittel: CLI injiziert Konventions-Präambel in den Prompt (aus CLAUDE.md generiert) und/oder AGENTS.md im Repo pflegen — sonst weichen Codex-Commits stilistisch ab (z.B. deutsche Kommentare)
  • Kein Streaming in Claudes Kontext: Zwischenstände nur via logs — für Task-Delegation unerheblich, Endbericht zählt
  • MCP-Bridge (Jobs als echte Tools): sauberere Evolution, aber v2 — Bash + JSON-CLI funktioniert sofort und ist trivial zu debuggen

08 · Risiken & Edge-Cases

Was schiefgehen kann — und was wir dagegen tun.

Event-Format-Drift
Größtes Risiko. codex exec --json-Events + Thread-ID-Extraktion sind das Fundament der Resume-Mechanik. Abbau: Slice 1 zuerst, Fixture-Tests aus echten Läufen, Parser tolerant (unbekannte Event-Typen überspringen, nie crashen), Codex-Versionscheck beim Spawn loggen.
Supervisor stirbt
Neustart/Crash → state.json sagt running, niemand lebt. Abbau: supervisorPid + Liveness-Check in jedem Leser (CLI list/status UND App-Monitor), Korrektur auf failed mit Grund. Muster existiert: healthCheck in BackgroundAgentLifecycle.
Parallel-Edits im selben Tree
Claude und Subagent editieren gleichzeitig in-place → Konflikte, vermischte Commits. Abbau: Worktree bleibt Opt-in (beschlossen), aber die Skill-Anleitung empfiehlt es für schreibende Jobs, während Claude selbst editiert. Rest ist bewusstes Restrisiko — wie bei nativen Subagents auch.
Report-Schema verletzt
Codex validiert via --output-schema selbst; falls doch Müll ankommt (Fence-Wrapping etc.): Fence-toleranter Parser (wie im Jarvis-Plan vorgesehen), sonst state=failed mit rohem Text in last-message.txt — nie stillschweigend verwerfen.
Doppelte Sichtbarkeit
Der Session-Scan (AgentScanCoordinator) findet die Codex-Session des Jobs auch in ~/.codex/sessions/ → Gefahr: gleiche Session zweimal in der Sidebar. Abbau: Indexer dedupliziert gegen codexThreadIDs aktiver Jobs; erst nach rm erscheint sie ggf. als normale externe Session.
send-Kollisionen
Zwei send-Aufrufe (oder App-Composer + CLI) gleichzeitig → zwei Resume-Turns auf einer Session. Abbau: state=running als Lock (Exit 3), Übergang spawning→running atomar; der Composer disabled sich auf running.
Bash-Timeout bei --wait
Claude Codes Bash-Tool killt lange synchrone Aufrufe (Default ~2 Min). Abbau: Skill-Regel — ab erwarteter Laufzeit >2 Min als Background-Task starten; das Harness benachrichtigt bei Prozessende von selbst.
Sandbox überrascht
git push, npm install etc. scheitern still an der Netzwerk-Sandbox → verwirrender Report. Abbau: Prompt-Präambel weist Codex an, Sandbox-Blocker explizit als openQuestion zu melden; --allow-network dokumentiert.
