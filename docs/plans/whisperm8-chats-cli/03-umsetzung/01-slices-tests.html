⚑ GO / NO-GO — NACH SLICE 1

Die riskanteste Annahme zuerst beweisen

  • Qualität: Beantwortet ein Chat mit overview + tail „was läuft, was braucht mich, was ist passiert?" inhaltlich korrekt — bei 5+ parallelen echten Sessions?
  • Status-Parität: CLI-Schätzung vs. App-Sidebar über ~20 reale Sessions — Abweichungen nur in den dokumentierten Fällen (PTY-Wissen, awaitingInput-Präzision)?
  • Latenz: Jeder Lese-Befehl < 2 s beim realen Workspace?
  • Ergonomie: Reichen die Refs (projekt/titel) im Alltag, oder braucht es Nachschärfung am Resolver?
  • No-Go-Antwort: Falls die Transcript-Schätzung zu oft daneben liegt → Slice 2 vorziehen (Live-Merge über den Socket wird Pflicht statt Bonus) — der Plan bleibt sonst identisch.
01 ✱ TESTPLAN

Unit, Integration, Abnahme.

Konvention wie im Bestand: Dependency Injection über Closures und kleine Protokolle, thematische Testdateien unter Tests/WhisperM8Tests/, Fixtures als Ressourcen. Neue Dateien: ChatsCLITests, SessionRefResolverTests, ChatsStatusProbeTests, AttentionModelBuilderTests, AgentControlServerTests, ChatsWaitEngineTests.

Unit · Pure Kerne

Resolver, Status, Attention

  • Resolver: alle 5 Stufen; Ambiguität; UUID-Präfix vs. Hex-Titel („cafebabe"); Umlaut-/Bindestrich-Normalisierung; @self ohne Env; archivierte außerhalb --all-Scope
  • StatusProbe: Fixture-Transcripts je Provider für working/awaitingInput/idle/errored; 30-s/120-s-Schwellen mit injiziertem now; fehlendes Transcript → unknown + availability; Background via Fake-Job-Verzeichnis
  • AttentionBuilder: Sortierung 1–5; 15-min-Fenster „frisch fertig"; Counts; leerer Workspace
  • Tail-Formatter: Turn-Gruppierung, Tool-Use-Einzeiler, --chars-Kürzung mit stderr-Vermerk
Unit · CLI-Vertrag

Parser, JSON, Exit-Codes

  • Argv-Parser je Subcommand inkl. ---Trenner, unbekannte Flags → Exit 1
  • list --json-Golden-File gegen das Paket-01-Schema (schemaVersion, live, counts, Capabilities)
  • Exit-Code-Mapping: RefError→3, Guard→4, Socket-Fehler→5, Timeout→124
  • Workspace-Reader: defekte JSON → Backup-Recovery-Pfad des Repository greift; UI-State-Read hat keine Schreib-Nebenwirkung (Datei-mtime unverändert!)
Unit · Socket

Server ohne App, Client ohne Server

  • NDJSON-Codec: Zeilengrenzen, 1-MiB-Limit, kaputtes JSON → error-Response
  • Send-Guards als pure Funktion (injizierte Fakes für Registry/Status): alle 5 Guards einzeln + Reihenfolge
  • Idempotenz: gleiche requestID → duplicate, Ablauf nach 60 s
  • Stale-Socket-Logik: EADDRINUSE + toter Lock → unlink+rebind; lebendiger Lock → Fehler
  • In-Process-Roundtrip: Server auf Temp-Socket + Client → ping/sessions.live/send-Konflikt
  • Token-Verifikation: gültig/falsch/fehlend → verified/unverified-Pfad
Unit · Wait

Event-Mechanik

  • Append an Fixture-Transcript → Event nach Debounce; Prädikat-Filter (attention vs. idle)
  • --since: älteres Event → sofortige Rückkehr; neueres → blockiert
  • delete/rename → Re-Arm auf neue Datei; size-Rückgang → Neu-Ableitung statt Fehler
  • Timeout → 124; SIGINT-Handler → 130
  • Bracketed-Paste-Wrapping: genau ein ESC[200~…ESC[201~-Block + optionales CR (String-Assertion am Fake-Terminal)
Manuelle Abnahme (nach S3/S4, durch dich)

Der Realitätscheck

  • 5+ parallele Sessions: overview stimmt mit der Sidebar überein; eine Session stoppen → wait feuert; tail erklärt das Ergebnis
  • Send-Kette aus Dialog A (Paket 04): tail → Rückfrage → send → Ziel-TUI zeigt einen Block, ein Submit; Audit-Eintrag vorhanden
  • Cross-Session-Szene: zwei Chats vergleichen, Folgeprompt an einen senden — komplett aus einem dritten Chat heraus
  • „Räum auf": rename/group/archive-Runde; App-UI reflektiert alles live; AgentSessions.json konsistent
  • App beenden → Lese-Befehle degradieren sauber; App starten → live: true kehrt zurück; Socket überlebt App-Neustart (Stale-Handling)
  • Ein sandboxed Codex-Subagent ruft chats list (ok) und chats send (sauberer Exit 5/4) auf
02 ✱ FAILURE-MATRIX

Wenn es schiefgeht, geht es definiert schief.

APP LÄUFT NICHT
Lesen: voll funktionsfähig, live: false, Capabilities konservativ. Handeln: Exit 5 < 2,5 s mit Fix-Hinweis. Skill degradiert auf Beobachten.
APP STÜRZT WÄHREND REQUEST
Verbindung reißt → CLI meldet Exit 5 „Antwort unvollständig". Für send gilt: Zustand unklar → CLI sagt das explizit („möglicherweise gesendet — Audit prüfen nach App-Neustart"), niemals „nicht gesendet" behaupten.
STALE SOCKET NACH CRASH
Neuer App-Start: EADDRINUSE → ping scheitert → lstat-Check → unlink + rebind. CLI vor App-Neustart: connect scheitert → Exit 5.
PROTOKOLL-MISMATCH
Alte CLI-Instanz trifft neue App (oder umgekehrt): protocolVersion-Check im Handshake → klare Meldung „App und CLI-Version passen nicht — whisperm8-Symlink prüfen", Exit 5.
TRANSCRIPT WEG/ROTIERT
Status unknown + availability-Feld; wait re-armt; kein Crash, kein Müll-Status. Kompaktierung (size schrumpft) → Neuableitung.
ZIEL-TUI IN SONDERZUSTAND
Permission-Dialog, Vim-Mode, Menü offen: Guards fangen das nicht vollständig — der Paste landet im Composer, das CR bestätigt schlimmstenfalls einen Dialog. Restrisiko dokumentiert; Mitigation: Default ifStatus awaitingInput,idle + Skill zeigt Prompt vorher. (Hook-Events präzisieren awaitingInput-Typ in v2.)
DOPPELTER SUPERVISOR
Zwei Chats senden ans gleiche Ziel: Serial-Queue im Server + Audit machen die Reihenfolge nachvollziehbar; zweiter Send sieht ggf. Status-Konflikt. Koordination bleibt Sache des Users (ein Supervisor-Chat als Konvention).
WORKSPACE-SCHEMA-DRIFT
Neuere App schreibt Felder, die die CLI-Version nicht kennt: lenienter Decoder (Bestand) ignoriert Unbekanntes — CLI bleibt funktionsfähig. Umgekehrt: CLI = App-Binary, gleiche Modelle, kein Drift im Normalfall.
RIESEN-WORKSPACE
100+ Sessions: Default-Scope (aktiv, 14 Tage) hält list klein; StatusProbe parallelisiert; wait deckelt bei 64 Watches mit Warnung.
KEIN WHISPERM8-KONTEXT
Aufruf aus normalem Terminal: alles funktioniert, actor: external im Audit, @self → Exit 3 mit Erklärung.
