Jarvis als CLI + Skill — kein UI, jeder Chat kann Supervisor sein.

Der Kern-Move

Der alte Jarvis-Plan hat ein Brain gebaut, weil codex exec kein Tool-Loop ist. Ein Claude-Code-Chat ist ein Tool-Loop — die CLI gibt ihm nur Augen und Hände. Damit wird die v2-Vision des alten Plans („Tools, damit auch externe Agents die Jarvis-Sicht nutzen") zur v1 — und das gesamte Brain-Kapitel entfällt.

01 ✱ TRENNUNG

Drei Schichten, klar getrennt.

Was die CLI ohne App kann, was nur die laufende App kann, und was der Skill daraus macht. Die Grenze zwischen Spalte 1 und 2 ist die wichtigste Architektur-Entscheidung des Plans.

LESE-PFAD ✱ APP-UNABHÄNGIG

Disk-Reads (immer verfügbar)

Die CLI ist das App-Binary — alle Reader-Services laufen in-process. Sie liest Workspace-JSON read-only, parst Transcripts und leitet Runtime-Status selbst ab.

  • list · overview · show · tail · wait · log
  • Quellen: AgentSessions.json, agent-ui-state.json, ~/.claude, ~/.codex, Jobs
  • Status-Ableitung one-shot über die pure AgentSessionStateMachine
  • Funktioniert auch bei geschlossener App
LIVE-PFAD ✱ NUR MIT APP

Control-Socket (die Hände)

Alles, was PTYs berührt oder den Workspace mutiert, geht über einen Unix Domain Socket in der App. Die App bleibt Single Writer — die CLI schreibt nie selbst an AgentSessions.json.

  • send · open · new · rename · group · archive
  • Socket: control.sock · JSON-Request/Response · 0600
  • Guards serverseitig: kein Send bei working, kein Selbst-Send, kein Archive bei laufender PTY
  • App zu? Klarer Fehler + Exit-Code, Lese-Pfad läuft weiter
SKILL ✱ VERHALTEN & AUTONOMIE

whisperm8-chats (die Regeln)

Der Skill macht aus Befehlen Verhalten: wann welcher Befehl, wie adressieren, wie berichten — und vor jeder mutierenden Aktion die explizite Rückfrage beim User.

  • Supervisor-Loop: overview → berichten → wait → reagieren
  • Vor send/archive: AskUserQuestion bzw. aktive Rückfrage — keine App-Notification nötig
  • Trigger: „was machen meine Chats", „sei mein Jarvis", „schick an …"
  • Empfohlene Allowlist: Lese-Befehle frei, Mutationen gated
02 ✱ DATENFLUSS

Wie ein Befehl durch das System läuft.

Vom Bash-Call im Chat bis zur Ziel-Session — mit der Weiche zwischen Disk-Read und Socket nach dem Parsen.

INPUT ✱ IRGENDEIN CHAT

Claude-Code-Session ruft die CLI

Bash-Call im Tool-Loop, gated durch Claude-Code-Permissions. Die PTY-Env enthält WHISPERM8_SESSION_ID — die CLI weiß, wer fragt.

argv → AgentChatsCLICommand · RefResolver löst „wolanin/seo" auf
PROCESS ✱ CLI-PROZESS (= APP-BINARY)

Parsen, auflösen, Weiche stellen

Workspace read-only laden, SessionRefResolver matcht die Referenz, dann: Lese-Befehl → Disk, Mutation/Live → Socket.

PFAD A ✱ DISK (APP EGAL)

Read + Status-Ableitung

AgentSessions.json, Transcript-Tail via Reader, Status one-shot aus mtime + JSONL-Tail. Für wait: kqueue/FSEvents auf die Transcript-Dateien.

PFAD B ✱ SOCKET (APP LÄUFT)

control.sock → App

JSON-Request mit actor (Session-ID des Aufrufers). App validiert Guards erneut, führt auf dem MainActor aus, antwortet mit Ergebnis oder Fehler.

App: AgentTerminalRegistry / AgentSessionStore / Launch-Service
OUTPUT ✱ WIRKUNG + ANTWORT

Ziel-Session reagiert, CLI antwortet strukturiert

Prompt landet als Bracketed Paste in der Ziel-PTY / Workspace ist mutiert / Tab geöffnet. stdout: Text oder −−json; Exit-Codes maschinenlesbar; Audit-Zeile geschrieben.

Querschnitt

Identität & Audit überall

Jeder mutierende Request trägt den actor aus WHISPERM8_SESSION_ID. Damit: Selbst-Send-Schutz, „(du)"-Markierung in list, und ein Audit-Log (chats-audit.jsonl), das chats log anzeigt.

Konventionen geerbt vom bestehenden whisperm8 agent: Ergebnis auf stdout, Fortschritt auf stderr, −−json-Objekte, Exit-Codes aus AgentCLIExit.

05 ✱ ENTSCHIEDEN

Bereits festgelegt.

Aus der Beratung vom 2026-07-19 — diese Punkte stehen und werden im Plan nicht erneut geöffnet.

01

Namespace: whisperm8 chats

Beschreibend statt Codename. Kein jarvis-Alias in v1 — kann später kommen, wenn gewünscht. agent bleibt den Codex-Subagent-Jobs vorbehalten.

ENTSCHIEDEN2026-07-19
02

Send-Autonomie über den Skill, nicht über die App

Der Skill schreibt vor: vor send/archive AskUserQuestion bzw. aktive Rückfrage im Chat. Keine Bestätigungs-Notification in der App — das Claude-Code-Permission-System ist das zweite Netz.

ENTSCHIEDEN2026-07-19
03

Kein MCP-Server

CLI + Skill reichen. Der Socket ist trotzdem so geschnitten, dass ein MCP-Wrapper später möglich bliebe — aber er ist ausdrücklich nicht Teil dieses Plans.

ENTSCHIEDEN2026-07-19
04

Kein Jarvis-UI

Inspector-Panel, Board-Views, Action-Queue-UI aus Rev 2 entfallen ersatzlos. Der alte Plan bleibt als Referenz unter docs/plans/jarvis-supervisor/ liegen und wird als „abgelöst durch chats-CLI" markiert.

ENTSCHIEDEN2026-07-19
05

Send-Regeln (Alignment-Runde)

Freigaben gelten pro Ziel-Session und pro Konversation, nie persistent. Alle drei Send-Arten sind erlaubt — Antworten, neue Aufträge, Steering via --if-status working — jeweils hinter dem Bestätigungs-Gate.

ENTSCHIEDEN2026-07-19
06

Kennzeichnung + Ein-Hop-Regel

Die App stellt jedem Agent-Send automatisch eine Marker-Zeile voran ([via whisperm8 chats · von … · HH:MM]). Skill-Regel: Auf via-chats-Prompts wird nie eigenständig weitergesendet — Ketten enden nach einem Hop.

ENTSCHIEDEN2026-07-19
07

Agent-Dialog über Transcript

Jarvis führt Dialoge via send → wait → tail — schleifensicher, null neue Mechanik. Ein Push-Rückkanal (Ziel-Agent promptet aktiv zurück) ist v2-Kandidat mit Loop-Guards.

ENTSCHIEDEN2026-07-19
08

Grenzen & Ergonomie

new aus Eigeninitiative: vorschlagen ja, starten nur nach Bestätigung. Aufräumen: eine Batch-Bestätigung (Multi-Select, Archives markiert). Lesen: alle Projekte, kein Scoping. Supervisor-Loop läuft, bis du stoppst. Destruktives Maximum ist archive — kein delete, kein PTY-Kill, keine Settings.

ENTSCHIEDEN2026-07-19
06 ✱ OFFENE PUNKTE

Was noch zu klären ist.

Nichts davon blockiert Slice 1. Die Punkte sind den Paketen zugeordnet, in denen sie detailliert sind.

01

Interrupt-Befehl (chats interrupt)?

Ein gezieltes ESC an eine arbeitende Session (kein generisches send-keys). Das GPT-Review argumentiert dafür; spezifiziert ist es in Paket 01. Empfehlung: nach Slice 2 mit Send-Erfahrung entscheiden.

VORSCHLAG: V2Paket 01
02

Wie viel darf new?

V1-Scope: neue Session als Tab in der App (Projekt + optionaler Initial-Prompt). Background-Agents (claude --bg) kann der Agent heute schon direkt spawnen — bleibt außerhalb.

SCOPE-CHECKPaket 01
03

Status-Ableitung: Parität CLI ↔ App

Die CLI leitet Status one-shot ab, die App event-driven. Bei Abweichungen gewinnt die App (sie sieht PTY-Zustand). Genaue Heuristik + Abgleich-Test in Paket 02; Parität ist Go/No-Go-Kriterium.

IN ARBEITPaket 02
04

Socket-Protokoll-Version

App und CLI sind dasselbe Binary — aber nach einem App-Update kann eine alte CLI-Instanz (Symlink zeigt auf neues Binary: unkritisch; laufende alte Prozesse: möglich) auf einen neuen Server treffen. Ein version-Feld im Handshake reicht.
