00 ✱ ÜBERSICHT

Die Befehlsmatrix auf einen Blick.

Zwei Klassen: Disk-Befehle laufen immer (App egal), Socket-Befehle brauchen die laufende App. overview ist ein Alias von list --sort attention --format board — dieselbe Logik, dasselbe JSON-Schema.

Registrierung im Binary

Neues Erstes-Token chats muss an zwei Stellen registriert werden: CLIModeDetector.recognizedCommands (CLIEntryPoint.swift:30) und der CLICommand.run-Switch (CLIEntryPoint.swift:80) → Dispatch an neues AgentChatsCLICommand. Konventionen 1:1 vom bestehenden agent-Namespace geerbt: Ergebnis auf stdout, Diagnose auf stderr (CLIIO), manuelle argv-Parser pro Subcommand (Muster AgentCLIArguments.swift), --json-Flag je Befehl.

01 ✱ VERTRAG

Exit-Codes, JSON-Regeln, Adressierung.

Ein Agent muss aus Exit-Code + stdout allein den nächsten Schritt ableiten können. Deshalb ist der Vertrag strenger als bei menschzentrierten CLIs.

EXIT 0 · OK
Erfolg. Bei wait: Prädikat erfüllt, Event auf stdout.
EXIT 1 · USAGE
Argument-/Flag-Fehler. Konsistent mit AgentCLIExit.usage des agent-Namespace (nicht grep-Konvention 2 — Repo-Konsistenz gewinnt).
EXIT 3 · NOT_FOUND
Referenz nicht auflösbar oder mehrdeutig. Bei Mehrdeutigkeit: Kandidatenliste auf stderr (und in --json als candidates), damit der Agent präzisieren kann.
EXIT 4 · CONFLICT
Precondition verletzt: Ziel working bei send, Selbst-Send, Archive bei laufender PTY, Rename bei manuellem Titel ohne --force, Drift zwischen Proposal und Ausführung.
EXIT 5 · APP_UNREACHABLE
Socket-Befehl, aber App läuft nicht / Socket tot. stderr nennt den Fix („WhisperM8 starten"). Lese-Befehle liefern stattdessen live: false und Exit 0.
EXIT 124 · TIMEOUT
Nur wait: Timeout abgelaufen, ohne dass das Prädikat eintrat. Kein Fehler im engeren Sinn — der Supervisor-Loop behandelt das als „nichts passiert, weiter beobachten".
EXIT 130 · INTERRUPTED
SIGINT während wait/laufendem Befehl.
JSON-Regeln

Jedes --json-Objekt

  • schemaVersion: 1 in jedem Top-Level-Objekt — Schema-Drift erkennbar
  • Nur Daten auf stdout — Fortschritt, Warnungen, Hinweise immer stderr
  • Ein Objekt pro Aufruf; wait gibt genau ein Event-Objekt aus
  • Offline-Markierung: live: false + runtime.source: "transcriptEstimate" wenn ohne App-Abgleich (Details Paket 02)
  • Datumsfelder ISO-8601, konsistent mit AgentWorkspaceRepository
Adressierung

<ref> — ein Positional, vier Formen

  • UUID voll oder Präfix ≥ 8 Zeichen — immer exakt
  • projekt/titel-fragment — Fuzzy, muss eindeutig sein
  • titel-fragment — Fuzzy über alle Projekte, muss eindeutig sein
  • @self — die aufrufende Session (aus WHISPERM8_SESSION_ID)
  • Mehrdeutig = Exit 3, immer. Auch bei Lese-Befehlen — keine heuristische Zielwahl (tmux-Prinzip). Auflösungs-Algorithmus: Paket 02
02 ✱ LESE-BEFEHLE

Sehen — immer, App egal.

chats list

[--project P] [--status S] [--attention] [--all] [--sort attention|activity] [--format table|board] [--json]
DISKREAD
Verhalten

Lädt den Workspace read-only (AgentWorkspaceRepository.load), leitet pro Session den Runtime-Status one-shot ab (Paket 02) und gibt die Liste aus. Läuft die App, wird zusätzlich der Live-Status über den Socket gemerged (--live auto-Verhalten, Details Paket 03) — Capabilities wie canSend sind nur dann verlässlich.

Default zeigt aktive Sessions (status != archived, Aktivität < 14 Tage); --all hebt beides auf.

Flags
  • --project P — Filter per Projektname/Pfad-Fragment (eindeutig)
  • --status S — working|awaitingInput|idle|stopped|errored
  • --attention — nur Kategorien „braucht dich" + „frisch fertig"
  • --sort — attention (Board-Reihenfolge) oder activity (Default)
  • --format board — gruppierte Board-Darstellung (= overview)
JSON-Schema (vollständig — gilt identisch für overview)
// chats list --json · schemaVersion 1 · ein Objekt { "schemaVersion": 1, "generatedAt": "2026-07-19T15:02:11Z", "live": true, // true = App-Status über Socket gemerged "self": "C41A…", // Session-ID des Aufrufers oder null "counts": { "needsYou": 2, "freshDone": 1, "working": 2, "idle": 3 }, "sessions": [{ "id": "6F2B41A0-…", "shortID": "6f2b41a0", "project": { "name": "wolanin", "path": "/Users/…/wolanin-md" }, "title": "seo-behandlungsseiten", "group": null, "provider": "claude", "kind": "chat", "status": "running", "runtime": { "status": "idle", "source": "app", // app | transcriptEstimate | supervisorJob "since": "2026-07-19T15:00:03Z", "revision": 2184533 }, // = Transcript-Bytes "attention": { "category": "freshDone", "rank": 3 }, "capabilities": { "canSend": true, "canOpen": true, "isAttachedPTY": true }, "isSelf": false, "lastActivityAt": "…", "lastAssistantLine": "Alle 6 Behandlungsseiten sind als Draft…", "externalSessionID": "a1b2…", "backgroundShortID": null, "transcript": { "path": "~/.claude/projects/…jsonl", "sizeBytes": 2184533, "availability": "available" } // missingExternalSessionID | missingTranscript | unsupported }] }

chats overview

[--json]
DISKREAD
Verhalten

Reiner Alias für list --sort attention --format board. Board-Gruppen: BRAUCHT DICH (awaitingInput, errored) → FRISCH FERTIG (working→idle/stopped, jünger als 15 min) → ARBEITET → IDLE (eingeklappt, Anzahl + Hinweis auf --all). Pro Zeile: Symbol+Farbe, projekt/titel, Status, „seit N min", Tier-1-Einzeiler (letzte Assistant-Zeile, 80 Zeichen), (du)-Markierung.

Warum Alias statt eigener Befehl
  • Ein JSON-Schema für beide — der Agent lernt eine Struktur
  • Eine Sortier-/Kategorisierungslogik (AttentionModelBuilder, Paket 02) — kein Drift zwischen zwei Implementierungen
  • overview bleibt als Wort im Skill, weil es die Supervisor-Absicht trifft

chats show

<ref> [--json]
DISKREAD
Verhalten

Volle Detailansicht einer Session: alles aus dem list-Eintrag plus model, reasoningEffort, claudeProfileName, createdAt, lastTurnAt, summary, Transcript-Pfad + Größe, Background-Felder (backgroundShortID, Supervisor-state), Fork-Herkunft. Für Background-Sessions wird SupervisorJobReader.readSingle gemerged.

Output / Fehler
  • Text: Key-Value-Blöcke, gruppiert (Identität / Laufzeit / Transcript / Provider)
  • --json: list-Session-Objekt + detail-Block
  • Ref mehrdeutig/fehlend → Exit 3 mit Kandidaten

chats tail

<ref> [--turns N] [--chars N] [--raw] [--json]
DISKREAD
Verhalten

Liest den Transcript-Tail über das bestehende Routing (AgentChatTailExtractor-Logik): .chat/.subagentJob → Claude-/Codex-Reader via externalSessionID; .backgroundChat → Supervisor-Job → linkScanPath; .agentView → zuletzt aktiver Job. Ausgabe als lesbare Turns: [user · HH:MM] / [assistant · HH:MM], Tool-Use als Einzeiler (⏺ Bash: swift test …), Thinking ausgelassen.

Defaults: --turns 3, hartes Limit --chars 6000 (Kontextbudget des lesenden Agenten schonen). Reader streamen; 50-MB-Transcripts sind über readTail (256 KB Tail) sicher.

Flags / Fehler
  • --turns N — letzte N User→Assistant-Runden
  • --chars N — Zeichen-Deckel, kürzt am Anfang, vermerkt Kürzung auf stderr
  • --raw — ungekürzte letzte Assistant-Message (für Zitate)
  • --json — Turns als Array strukturierter Messages
  • Kein Transcript → Exit 0 mit availability-Hinweis (kein Fehler: die Info „noch kein Transcript" ist eine gültige Antwort)

chats wait

[--ref R]… [--until attention|idle|statusChange] [--since REV] [--timeout SEC] [--json]
DISKREAD
Verhalten

Blockiert, bis das Prädikat eintritt, und gibt genau ein Event aus — der Baustein für den Supervisor-Loop ohne Polling-Spam. Ohne --ref werden alle aktiven Sessions beobachtet; mehrere --ref sind erlaubt. Mechanik (kqueue auf Transcript-Dateien + Hook-Event-Files, Re-Evaluation per Status-Decider): Paket 02.

--since REV (Transcript-Revision aus einem früheren list) verhindert die Lücke zwischen zwei wait-Aufrufen: Liegt seit REV bereits ein passendes Event vor, kehrt wait sofort zurück.

Prädikate / Exit
  • --until attention (Default) — irgendeine Session wird awaitingInput/errored oder beendet einen Turn
  • --until idle — Ziel-Session(s) beenden ihren Turn
  • --until statusChange — jeder Übergang
  • --timeout SEC — Default 1800; Exit 124 ohne Event
  • Event-JSON: {event, session{…}, from, to, afterSeconds, revision, source}

chats audit

[--limit N] [--session R] [--json]
DISKREAD
Verhalten

Zeigt das Aktionsprotokoll aus chats-audit.jsonl (geschrieben ausschließlich von der App bei Socket-Mutationen — Details Paket 03). Eine Zeile pro Aktion: Zeit, Actor (projekt/titel oder „extern"), Befehl, Ziel, Ergebnis.

Zweck
  • Nachvollziehbarkeit, wenn mehrere Chats gleichzeitig managen
  • Debugging: „warum hat Session X diesen Prompt bekommen?"
  • Kein Prompt-Volltext im Log — nur Länge + erste 80 Zeichen (Privacy-Default)
03 ✱ LIVE-BEFEHLE

Handeln — über den Socket.

Alle Befehle hier sprechen mit der laufenden App (Protokoll: Paket 03). Die Guards laufen in der App, atomar — nicht in der CLI. Zwischen „CLI prüft Status" und „Prompt landet in der PTY" darf kein Fenster für Überraschungen liegen.

chats send

<ref> "<prompt>" [--if-status S,S] [--no-submit] [--force] [--json]
SOCKETMUT
Verhalten

Schickt den Prompt an die Ziel-PTY. Die App validiert in einem kritischen Abschnitt auf dem MainActor: Controller existiert + isRunning → Runtime-Status ∈ --if-status (Default awaitingInput,idle) → Ziel ≠ Actor. Erst dann: Bracketed-Paste-Wrap (ESC[200~ … ESC[201~), Senden via sendUserText, nach ~80 ms ein CR (0x0d) — außer bei --no-submit. Audit-Zeile immer.

Kennzeichnung (entschieden 2026-07-19): Die App stellt dem Prompt automatisch eine Marker-Zeile voran — [via whisperm8 chats · von <projekt/titel|extern> · HH:MM]. Die Empfänger-Session erkennt agentische Prompts; zusammen mit der Ein-Hop-Regel im Skill enden Agent-Ketten nach einem Hop.

Neu zu bauen: Bracketed Paste + Auto-Submit existieren heute nicht (sendUserText sendet bewusst ohne Enter) — das ist Teil von Slice 2, als eigene Methode sendPrompt(_:submit:) am AgentTerminalController.

Flags / Antwort / Fehler
  • --if-status working — bewusst in laufende Arbeit senden (z. B. Steer-Message); Default verbietet das
  • --force — überstimmt den Status-Guard, nie den Selbst-Send-Schutz
  • --no-submit — Text nur in den Composer legen, User schickt ab
  • Antwort-JSON: {ok, ack: "delivered", target{id,title}, promptChars, requestID}
  • Idempotenz: CLI generiert requestID; App dedupliziert 60 s — ein Timeout-Retry pastet nie doppelt
  • Status-Konflikt → Exit 4 (+ aktueller Status im JSON) · App zu → Exit 5

chats open

<ref> [--json]
SOCKETUI
Verhalten

Selektiert/öffnet die Session in der App über den bestehenden Fokus-Pfad WindowRequestCenter.requestSessionFocus(sessionID:) — derselbe, den Notification-Klicks nutzen (Ziel-Fenster-Auflösung, Projekt-Gruppe aufklappen, Stale-Guard inklusive). Bringt die App nach vorn.

Fehler
  • Session archiviert/fehlt → Exit 3 (Stale-Guard der App meldet es zurück)
  • App zu → Exit 5

chats new

--project <pfad|name> [--provider claude|codex] [--title T] [--prompt "…"] [--json]
SOCKETMUT
Verhalten

Legt über die App eine neue Session an: upsertProject → createSession(provider:…, initialPrompt:, shouldLaunchOnOpen: true) → Tab öffnet, PTY startet beim Rendern (bestehender Auto-Start-Pfad). Der bestehende AgentChatLaunchService (heute nur Codex) wird um den Claude-Fall verallgemeinert. Antwort erst nach Persistenz, mit der neuen Session-ID — der Agent kann sofort wait --ref darauf setzen.

Flags / Grenzen
  • --project — Pfad (muss existieren) oder eindeutiger Workspace-Projektname
  • --provider — Default claude
  • --prompt — Initial-Prompt, läuft über den regulären Launch-Flow
  • Nicht in v1: Background-Agents (claude --bg kann der Agent direkt aufrufen), Worktrees, Profile-Auswahl
  • Antwort: {ok, session{id, title, project}}

chats rename · group · archive

<ref> "<wert>" | --clear | [--force] [--json]
SOCKETMUT
Verhalten

Workspace-Mutationen laufen durch den AgentSessionStore der App — die CLI schreibt nie selbst an AgentSessions.json (Single-Writer, debounced Persistenz, Backups bleiben konsistent). Validierung wird vor dem mutate-Aufruf gehoisted (Store-Lock-Regel).

  • rename <ref> "Titel" — setzt titleIsAutoGenerated = false
  • group <ref> "Gruppe" / group <ref> --clear
  • archive <ref> — Status → .archived
Guards (Exit 4 wenn verletzt)
  • rename: manuell gesetzter Titel (titleIsAutoGenerated != true) nur mit --force
  • archive: nie bei runtime == working oder laufender PTY; --force terminiert nicht — es archiviert nur eine idle Session mit noch offener PTY
  • Drift-Check: App validiert gegen den aktuellen In-Memory-Zustand (autoritativ), nicht gegen den Disk-Stand der CLI
  • Alle drei: Audit-Zeile mit Vorher/Nachher
Offen zur Abnahme: chats interrupt

Das GPT-Review argumentiert: Ein Supervisor ohne Abbruchmöglichkeit kann außer Kontrolle geratene Sessions nicht managen. Spezifiziert wäre es klein: interrupt <ref> --if-status working sendet genau ein ESC (Claude-TUI-Abbruch) über denselben atomaren Socket-Pfad wie send — kein generisches send-keys. Empfehlung: erst nach Slice 2 entscheiden, wenn Erfahrungswerte mit send vorliegen. Bis dahin: nicht gebaut.

04 ✱ TERMINAL

Vier Sessions, echte Ausgaben.

So sieht der Vertrag in der Praxis aus — inklusive der Fehlerfälle, die ein Agent sauber behandeln muss.

TERMINAL A

list — Tabelle mit Selbst-Markierung

Exit 0

Die Standard-Tabelle: kompakt, eine Zeile pro Session, Capabilities als Kürzel. Die aufrufende Session erkennt sich selbst.

chats list
❯ whisperm8 chats list --project whisperm8 REF PROJEKT/TITEL STATUS SEIT CAP 6f2b41a0 whisperm8/pill-redesign awaitingInput 4 min S·O·P 9c11d3e2 whisperm8/chats-cli-plan working 18 min S·O·P (du) b04771fa whisperm8/statusline-feature idle 2 h –·O·– 3 Sessions · live: ja (App verbunden) · S=send O=open P=pty
Lesart

Capabilities-Spalte

  • S·O·P — sendbar, öffenbar, PTY läuft in der App
  • –·O·– — indizierte externe Session: Tab öffenbar, aber keine laufende PTY → send würde Exit 4 liefern; der Agent sieht das vorher
  • Die letzte Zeile (stderr) nennt live-Zustand — bei geschlossener App: „live: nein (Status geschätzt)"
Interactions

Typische Folge-Befehle

  • chats show 6f2b41a0 — Details zur wartenden Session
  • chats tail 6f2b41a0 --turns 1 — was fragt sie?
  • chats send 6f2b41a0 "…" — antworten (nach User-Rückfrage!)
TERMINAL B

Mehrdeutige Referenz — Exit 3

Exit 3 · NOT_FOUND

Fuzzy darf bequem sein, aber nie raten. Der Fehler liefert die Kandidaten gleich mit — der Agent präzisiert im nächsten Call.

chats send — mehrdeutig
❯ whisperm8 chats send seo "Bitte Status-Update" Fehler: „seo" ist mehrdeutig (2 Treffer): 6f2b41a0 wolanin/seo-behandlungsseiten idle 3e88c210 rankm8/seo-audit-juli working Präzisiere: wolanin/seo oder UUID-Präfix. exit: 3
TERMINAL C

send-Guards: Konflikt statt Überraschung

Exit 4 · CONFLICT

Ziel arbeitet gerade — Default verweigert. Der Agent entscheidet bewusst: warten (wait) oder steuern (--if-status working).

chats send — Status-Guard
❯ whisperm8 chats send marketing/creative "Stopp, andere Priorität" Konflikt: marketing/creative-batch ist working (seit 31 min). Optionen: warten (chats wait --ref marketing/creative --until idle) oder bewusst reinsenden: --if-status working exit: 4 ❯ whisperm8 chats send marketing/creative "Stopp, andere Priorität" --if-status working --json {"schemaVersion":1,"ok":true,"ack":"delivered","target":{"id":"88a2…", "title":"creative-batch"},"promptChars":24,"requestID":"f4e1…"}
Warum so

Der Guard ist Ergonomie, kein Papierkram

  • Ein Prompt mitten in einen laufenden Turn ist meist ein Versehen — Default schützt davor (TOCTOU-sicher, weil die App atomar prüft, nicht die CLI)
  • --if-status working macht die Ausnahme explizit und dokumentiert — genau das Muster, das der Skill dem Agenten vorschreibt: erst User fragen, dann Flag setzen
  • Selbst-Send (@self oder eigene UUID als Ziel) ist immer Exit 4 — auch mit --force: die Endlosschleife Agent→eigene PTY→Agent ist der eine nicht verhandelbare Guard
TERMINAL D

App zu: Lesen degradiert, Handeln bricht sauber

Exit 0 / Exit 5

Der wichtigste Unterschied der zwei Befehlsklassen, einmal nebeneinander.

App nicht gestartet
❯ whisperm8 chats overview Hinweis: WhisperM8 läuft nicht — Status aus Transcripts geschätzt, PTY-Infos fehlen. ── BRAUCHT DICH ─────────────────────────── 1 ── ◆ whisperm8/pill-redesign awaitingInput seit 9 min · geschätzt ── ARBEITET ─────────────────────────────── 1 ── ▶ marketing/creative-batch working seit 44 min · geschätzt ❯ whisperm8 chats send whisperm8/pill "Variante B bitte" Fehler: WhisperM8-App nicht erreichbar (control.sock fehlt). Starte die App, dann erneut versuchen. Lese-Befehle funktionieren weiter. exit: 5
05 ✱ EDGE-CASES

Was der Vertrag abfangen muss.

SELBST-SEND
Ziel == Actor (WHISPERM8_SESSION_ID): immer Exit 4, kein --force-Override. Verhindert die Prompt-Endlosschleife.
SESSION OHNE PTY
Indexierte/extern gestartete Session ohne App-Terminal: canSend: false in list/show; send → Exit 4 mit Hinweis „Tab öffnen mit chats open, dann startet die PTY".
DOPPELTER SEND (RETRY)
CLI-Timeout nach erfolgreichem Paste: Retry mit derselben requestID wird von der App dedupliziert (60-s-Cache) → ack: "duplicate", kein zweiter Paste. Niemals „Timeout = nicht gesendet" annehmen.
PROMPT MIT NEWLINES
Bracketed Paste garantiert: mehrzeiliger Prompt landet als ein Block, submitted genau einmal (CR nach Delay). Ohne Wrap würde jede Zeile sofort abgeschickt — deshalb ist der Wrap Pflicht, nicht Option.
ARCHIVIERTE ZIELE
send/open/rename/group auf archivierte Session → Exit 3 („nicht gefunden" aus Sicht aktiver Refs); --all-Scope gilt nur für list/show/tail.
WORKSPACE-DEBOUNCE
Die App schreibt debounced (0,5 s) — der Disk-Read der CLI kann Sekunden alt sein. Für Lese-Befehle akzeptiert (Anzeige), für Mutationen irrelevant (App validiert gegen In-Memory-Zustand).
KEIN EXTERNAL-ID
Frisch erstellte Session ohne externalSessionID: tail → availability: missingExternalSessionID, Status unknown; kein Fehler-Exit.
PROFIL-SESSIONS
Transcript-Roots umfassen ~/.claude/projects und ~/.claude-profiles/<name>/projects (bestehender AgentTranscriptLocator); claudeProfileName steuert die Auflösung. Worktree-cwds werden via canonicalProjectPath normalisiert.
JSON IN PROMPTS
Prompts mit Quotes/Sonderzeichen: Positional nach ---Trenner möglich (chats send ref -- "…", Parser-Muster aus AgentCLIArguments übernommen).
AGENT-KETTEN
Chat A sendet an B, B hat denselben Skill: B sieht die Marker-Zeile und darf laut Skill-Regel nie eigenständig weitersenden (Ein-Hop). Bs Antwort liest A über wait + tail aus dem Transcript — ein Push-Rückkanal ist bewusst v2 (Paket 05).
