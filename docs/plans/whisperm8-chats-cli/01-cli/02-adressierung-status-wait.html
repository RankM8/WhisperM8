PAKET 02 ✱ PURE KERNE ✱ v1

Auflösen, ableiten, warten.

Die drei puren, unit-testbaren Kerne hinter dem Befehlsvertrag: der SessionRefResolver, die One-Shot-Status-Ableitung aus dem Transcript und das Attention-Modell — plus die Event-Mechanik, mit der wait ohne App und ohne Polling blockiert.

Neu (pur)SessionRefResolver · ChatsStatusProbe · AttentionModelBuilder · ChatsWaitEngine
WiederverwendetAgentTranscriptLocator · Parser · StatusDecider · Reader · SupervisorJobReader
MainActor nötigNichts — alles nonisolated/static
TestsFixture-Transcripts + Golden-Outputs
00 ✱ ÜBERSICHT

Warum diese drei Kerne pur sein müssen.

Alles in diesem Paket läuft im CLI-Prozess ohne App, ohne Store-Registry, ohne MainActor — und ist damit vollständig mit Fixtures testbar. Die Recherche hat bestätigt: die Bausteine existieren bereits als nonisolated static-Funktionen im Repo.

Input

Was reinkommt

  • Workspace: AgentWorkspaceRepository(fileURL:).load() — reine Disk-Funktion mit Backup-Recovery. Nicht AgentSessionStore.loadWorkspace() (liest aus der In-Memory-Registry)
  • UI-State: direkter JSONDecoder auf agent-ui-state.json. Nicht loadUIState() — die migriert und persistiert (!)
  • Transcripts: AgentTranscriptLocator + Reader (Claude/Codex/Supervisor)
  • Umgebung: WHISPERM8_SESSION_ID für @self/isSelf
Output

Was rausgeht

  • Aufgelöste Session (oder Exit-3-Fehler mit Kandidaten)
  • Runtime-Status + source + revision pro Session
  • Board-Struktur (Kategorien, Ränge, Counts) für list/overview
  • wait-Event (genau eines) mit from/to/afterSeconds
01 ✱ SESSIONREFRESOLVER

Bequem für Menschen, exakt für Maschinen.

tmux-Prinzip: Skripte adressieren nie heuristisch. Fuzzy ist erlaubt, aber nur wenn das Ergebnis eindeutig ist — sonst Exit 3 mit Kandidaten. Für Lese- und Schreib-Befehle gilt derselbe Resolver, dieselbe Strenge.

STUFE 1 · @SELF
@self → Session-UUID aus WHISPERM8_SESSION_ID. Fehlt die Variable (Aufruf außerhalb einer WhisperM8-PTY): Exit 3 mit Erklärung.
STUFE 2 · UUID EXAKT
Voll-UUID (case-insensitiv) → exakter Match auf session.id. Kein Fallthrough zu Fuzzy.
STUFE 3 · UUID-PRÄFIX
Hex-String ≥ 8 Zeichen (Bindestriche optional) → Präfix-Match auf der UUID. Genau ein Treffer nötig; mehrere → Exit 3. Kürzere Präfixe werden als Titel-Fragment behandelt (Stufe 4/5).
STUFE 4 · PROJEKT/TITEL
projekt-fragment/titel-fragment — beide Seiten normalisiert (lowercase, Diakritika gefaltet, Nicht-Alphanumerik → Trenner). Projekt matcht auf Name oder letztes Pfadsegment; Titel als Substring-, dann Subsequenz-Match. Eindeutig oder Exit 3.
STUFE 5 · TITEL GLOBAL
Ohne /: Fragment gegen alle aktiven Sessions (Titel, dann Gruppenname). Eindeutig oder Exit 3 mit Kandidatenliste (max. 5, nach Aktivität sortiert).
SCOPE
Default: aktive Sessions (status != .archived). list/show/tail --all erweitern auf archivierte; Mutations-Befehle sehen archivierte nie (Paket 01, Edge „archivierte Ziele").
API

Purer Kern, injizierbar

  • SessionRefResolver.resolve(ref: String, in: [AgentChatSession], projects: [AgentProject], selfID: UUID?) → Result<AgentChatSession, RefError>
  • RefError: .notFound, .ambiguous(candidates: [Candidate]), .noSelfContext — mappt 1:1 auf Exit 3 + stderr-Text
  • Keine I/O, keine Uhrzeit — vollständig deterministisch testbar
  • Ablage: WhisperM8/CLI/SessionRefResolver.swift (CLI-only, aber pur genug für spätere App-Nutzung)
Testfälle (Auszug, vollständig in Paket 05)

Was der Resolver beweisen muss

  • „seo" bei zwei SEO-Sessions → ambiguous mit beiden Kandidaten
  • „wolanin/seo" → eindeutig, obwohl global mehrdeutig
  • 8-Zeichen-Hex, das zufällig ein Wort ist („cafebabe") → UUID-Präfix gewinnt vor Titel-Match
  • Umlaute/Bindestriche: „behandlungsseiten" matcht „SEO-Behandlungsseiten"
  • @self ohne Env-Variable → .noSelfContext
02 ✱ STATUS-ONE-SHOT

Der Status kommt aus dem Transcript.

Die App leitet Runtime-Status event-driven ab — die CLI macht exakt dieselbe Ableitung one-shot, mit denselben puren Bausteinen. Kein Fork der Logik, kein Drift: gleiche Funktionen, gleiche Schwellen.

PRO SESSION

Session → Transcript-Pfad

AgentTranscriptLocator.locate: Claude <root>/<encoded-cwd>/<externalSessionID>.jsonl (Roots: ~/.claude/projects + ~/.claude-profiles/<name>/projects, cwd via canonicalProjectPath normalisiert); Codex rekursiv über ~/.codex/sessions/YYYY/MM/DD/; Background via SupervisorJobReader → linkScanPath.

kein Pfad → runtime.status = unknown, availability erklärt warum
STAT-FIRST

fileStat → mtime + size

Bestehendes AgentSessionRuntimeWatcher.fileStat (nonisolated static): 1 stat()-Syscall. size wird die revision im JSON-Output — monoton wachsend, vergleichbar zwischen CLI-Aufrufen und ideal für wait --since.

nur wenn nötig: 64-KB-Tail lesen (readTail)
PARSE + DECIDE

lastEvent → Decision

AgentTranscriptParser.lastEvent(in:provider:) (Rückwärts-Scan) → AgentTranscriptStatusDecider.decide(lastEvent:fileMTime:now:…). Schwellen aus dem Bestand: idle nach 30 s Ruhe, workingStall nach 120 s; tool_use/pause_turn zählen nicht als Turn-Ende.

Ergebnis + Quelle markieren
OUTPUT

runtime {status, source, since, revision}

source: "transcriptEstimate" — außer die App läuft und liefert über den Socket ihren autoritativen Status (source: "app", inkl. PTY-Wissen und Hook-Events). Background-Jobs: source: "supervisorJob" aus state.json.

Parität CLI ↔ App

Was der Schätzer nicht sehen kann

  • PTY-Existenz: ob ein Terminal in der App läuft, weiß nur die App → offline sind capabilities konservativ (canSend: false) und isAttachedPTY: null
  • Hook-Events: PermissionRequest-präzises awaitingInput kommt aus der Hook-Bridge; der Transcript-Schätzer erkennt awaitingInput nur aus dem letzten Event-Typ — dokumentierte Unschärfe
  • Regel: läuft die App, merged die CLI deren Status (ein Socket-Call sessions.live, Paket 03) — der Schätzer ist Fallback, nie Konkurrenz
  • Go/No-Go-Test: 20 reale Sessions, CLI-Schätzung vs. App-Anzeige — Abweichungen nur in dokumentierten Fällen
Performance

Budget: < 2 s für list bei 50 Sessions

  • stat-first: 50 stat()-Calls sind Rauschen; Tail-Reads nur für Sessions mit unklarem Zustand
  • Tail-Reads parallelisierbar (TaskGroup), Reader sind streamend und 50-MB-sicher
  • Kein Cache in v1 — ein CLI-Prozess lebt Millisekunden; agent-index-cache.json wird nicht angefasst
03 ✱ ATTENTION-MODELL

Sortiert nach „braucht mich".

Direkt aus dem Jarvis-Plan Rev 2 übernommen — dieselben fünf Kategorien, jetzt als CLI-Board statt SwiftUI-View. Der Builder ist pur: Sessions + Status rein, Board-Struktur raus.

1 · AWAITING INPUT
Session wartet auf den User (Frage, Permission-Prompt). Board-Gruppe BRAUCHT DICH, Symbol ◆ amber.
2 · ERRORED
Turn mit Fehler beendet / Prozess gestorben. BRAUCHT DICH, Symbol ✖ rot.
3 · FRISCH FERTIG
working → idle/stopped vor weniger als 15 min. Gruppe FRISCH FERTIG, Symbol ● grün. Kein persistiertes „gelesen"-Flag in v1 — das Zeitfenster ersetzt es (bewusste Vereinfachung gegenüber Rev 2).
4 · WORKING
Arbeitet. Gruppe ARBEITET, Symbol ▶ violett, „seit N min" aus Turn-Start.
5 · IDLE / REST
Alles andere (idle > 15 min, stopped, unknown). Im Board eingeklappt: nur Anzahl + --all-Hinweis; in list normal gelistet.
API

AttentionModelBuilder.build(sessions: [SessionWithRuntime], now: Date) → AttentionBoard — liefert Kategorien, Ränge und Counts; counts landet 1:1 im JSON-Header von list/overview. Innerhalb einer Kategorie: Sortierung nach Statuswechsel-Zeitpunkt, neueste zuerst. Pur, kein I/O, now injiziert.

04 ✱ WAIT-ENGINE

Blockieren ohne Polling, ohne App.

wait ersetzt das gesamte Report-/Notification-System des alten Plans. Die Mechanik kombiniert kqueue-File-Watching (Muster: ClaudeHookBridge) mit dem Status-Decider — und einem Sicherheitsnetz-Timer.

SETUP

Beobachtungsmenge bestimmen

Ohne --ref: alle aktiven Sessions mit auffindbarem Transcript. Mit --ref (mehrfach): genau diese. Initial einmal Status ableiten — erfüllt schon jetzt das Prädikat (oder liegt ein Event nach --since REV), sofort Exit 0 mit Event.

pro Transcript: DispatchSourceFileSystemObject (O_EVTONLY, .write/.extend/.delete/.rename)
EVENT-LOOP

Write → debounced Re-Evaluation

Bei jedem Write (300 ms Debounce pro Datei): stat → bei Änderung Tail lesen → lastEvent → decide → Statusübergang gegen den letzten bekannten Zustand prüfen. Zusätzlich für Claude-Sessions: kqueue auf die Hook-Event-Files (claude-session-events/<id>.jsonl, falls vorhanden) — liefert awaitingInput präziser und schneller als der Transcript-Schätzer.

delete/rename → Watch re-armen · Fallback-Timer alle 10 s (Missed-Event-Netz) · neue Sessions: Verzeichnis-Watch auf die Transcript-Roots
AUFLÖSUNG

Erstes passendes Event gewinnt

Prädikat erfüllt → genau ein Event-JSON auf stdout, Exit 0, alle Watches abbauen. Timeout → Exit 124 (stumm bei --json, Hinweis auf stderr). SIGINT → Exit 130. Die 30-s-idle-Schwelle des Deciders bedeutet: „Turn fertig" wird ~30 s nach der letzten Transcript-Zeile gemeldet — akzeptierte Latenz, dokumentiert im Skill.

TERMINAL

Der Supervisor-Loop in drei Zeilen

wait im Loop
❯ whisperm8 chats overview --json # Runde 1: Lagebild, revision je Session merken {"schemaVersion":1, "counts":{"needsYou":0,"working":3,…}, …} ❯ whisperm8 chats wait --until attention --timeout 1800 --json … blockiert 7 Minuten … {"schemaVersion":1,"event":"statusChanged", "session":{"id":"6f2b…","title":"pill-redesign","project":"whisperm8"}, "from":"working","to":"awaitingInput","afterSeconds":412, "revision":1948210,"source":"hookEvent"} ❯ whisperm8 chats tail 6f2b41a0 --turns 1 # was will sie? → dann berichten/handeln
Lückenlosigkeit

--since schließt das Fenster zwischen zwei waits

  • Zwischen Event-Ausgabe und dem nächsten wait-Aufruf können Statuswechsel passieren
  • Der Agent übergibt die höchste gesehene revision: wait --since 1948210 → liegt bereits ein neueres passendes Event vor, kehrt wait sofort zurück statt zu blockieren
  • Revision = Transcript-Bytes: billig zu erheben, monoton, pro Session eindeutig — kein globaler Event-Log nötig (der wäre v2, falls je gebraucht)
05 ✱ EDGE-CASES

Die unbequemen Fälle.

TRANSCRIPT ROTIERT
Claude kompaktiert/ersetzt JSONL-Dateien: .delete/.rename-Event → Locator erneut ausführen, Watch auf neue Datei re-armen (Muster aus AgentSessionRuntimeWatcher). Revision kann dabei schrumpfen → wait behandelt size-Rückgang als „neue Datei, Zustand neu ableiten", nie als Fehler.
SESSION OHNE EXTERNAL-ID
Frisch erstellt, noch kein Transcript-Schlüssel: Status unknown, in wait-Beobachtung ausgenommen; sobald der Verzeichnis-Watch die neue Datei sieht, wird nachgezogen.
ZWEI SESSIONS, EIN CWD
Mehrere Sessions im selben Projektordner: Transcript-Zuordnung läuft über externalSessionID im Dateinamen — kein Kollisionsrisiko. Nur die Stufe-2-Glob-Suche (cwd-Header-Verifikation) braucht den Header-Check aus dem Bestand.
CODEX-DATUMSGRENZE
Codex legt Transcripts unter YYYY/MM/DD an — ein über Mitternacht laufender Turn wechselt nie die Datei (Datei bleibt beim Start-Datum), aber der Locator-Scan muss rekursiv bleiben (ist er: Filename-Suffix-Match).
HOOK-FILE FEHLT
Hook-Event-Files existieren nur für App-gestartete Claude-Sessions. wait nutzt sie opportunistisch (source: "hookEvent"), fällt sonst transparent auf Transcript-Watching zurück — kein Feature-Unterschied, nur Präzision/Latenz.
UHRZEIT-ABHÄNGIGKEIT
Decider-Schwellen (30 s/120 s) hängen an now — im Test injiziert. Die CLI nutzt Wall-Clock; Zeitzonen egal (alles UTC-Vergleiche auf mtime).
VIELE SESSIONS
Watch-Menge = aktive Sessions (Default-Scope, kein --all): typisch < 20 FDs. kqueue-FD-Limit ist kein Thema; trotzdem Deckel bei 64 beobachteten Dateien mit stderr-Warnung.
