PAKET 04 ✱ SKILL ✱ v1

Aus Befehlen wird Verhalten.

Die CLI kann alles — der Skill entscheidet, wann sie was tut. Er macht jeden Chat auf Zuruf zum Beobachter, Zuarbeiter oder Dauer-Supervisor, und er kodiert die eine Regel, die du festgelegt hast: vor jeder mutierenden Aktion wird gefragt.

00 ✱ ÜBERSICHT

Ein Skill, drei Rollen.

Der Skill triggert auf natürliche Formulierungen und wählt selbst die passende Rolle — eskaliert aber nie eigenmächtig von Beobachten zu Handeln.

Trigger

Wann der Skill lädt

  • Beobachten: „was machen meine Chats", „läuft noch was?", „Status meiner Sessions", „wartet irgendwas auf mich?"
  • Zuarbeiten: „schick an Chat X …", „antworte dem Wolanin-Chat", „starte einen Chat in Projekt Y", „räum meine Sessions auf"
  • Supervisor: „sei mein Jarvis", „überwache meine Chats", „sag mir Bescheid, wenn was fertig ist"
  • Nicht triggern bei: Codex-Subagent-Jobs (codex-subagent-Skill), Transkription (whisperm8-transcription)
Abgrenzung

Was der Skill bewusst nicht tut

  • Keine eigene Statuslogik — er vertraut overview --json und interpretiert nur
  • Kein Berichts-Spam: Er fasst zusammen, statt rohe CLI-Tabellen zu zitieren
  • Kein --force und kein --if-status working aus eigenem Antrieb — nur wenn der User es in diesem konkreten Fall verlangt hat
  • Keine Dauerschleife ohne expliziten Supervisor-Auftrag
01 ✱ AUTONOMIE

Drei Stufen, klare Grenzen.

Die Entscheidung vom 2026-07-19: Send-Autonomie wird im Skill gelöst, nicht in der App. Konkret heißt das — zwei Verteidigungslinien: die Skill-Regel (fragen!) und das Claude-Code-Permission-System (Bash-Gate auf mutierende Befehle).

STUFE A · BEOBACHTEN
Default, keine Rückfrage nötig. list, overview, show, tail, audit, wait — alles Lesende läuft frei. Der Skill fasst Ergebnisse zusammen (max. 5 Zeilen für ein Standard-Lagebild) und nennt bei degradiertem Modus („App läuft nicht") die Einschränkung.
STUFE B · ZUARBEITEN
Mutation nur nach Bestätigung im konkreten Fall. Vor send: Prompt-Text wörtlich zeigen + Ziel nennen → AskUserQuestion (interaktiv) bzw. explizite Rückfrage im Chat-Text. Vor archive/rename --force: dasselbe. open und new gelten als leicht: kurz ankündigen reicht, wenn sie direkt aus dem User-Auftrag folgen; new aus Eigeninitiative wird vorgeschlagen (Projekt + Initial-Prompt zeigen) und erst nach Ja gestartet. Aufräum-Runden: eine Batch-Bestätigung per AskUserQuestion mit Multi-Select (Vorher→Nachher-Liste, Archives markiert) statt Einzelfragen.
STUFE C · SUPERVISOR
Nur nach explizitem Auftrag („sei mein Jarvis"). Loop: overview → berichten → wait → bewerten → berichten/vorschlagen. Auch hier: Sends brauchen Bestätigung — es sei denn, der User hat für eine konkrete Ziel-Session pauschal freigegeben („du darfst dem Wolanin-Chat direkt antworten"). Diese Freigabe gilt pro Session und pro Konversation, nie global, und wird im Bericht erwähnt („habe direkt geantwortet, wie freigegeben").
IMMER
Nie an @self senden (die CLI blockt es ohnehin — der Skill versucht es gar nicht erst). Exit 4/5 sauber erklären statt wegzuignorieren. Bei Exit 3 (mehrdeutig): Kandidaten dem User zeigen, nicht selbst raten.
02 ✱ SUPERVISOR-LOOP

Der Jarvis-Rhythmus.

Das Herzstück der Stufe C — als festes Muster im Skill, damit jede Session denselben verlässlichen Rhythmus fährt.

# Supervisor-Loop (SKILL.md-Auszug, Pseudocode) 1. LAGEBILD overview --json → kompakt berichten (needsYou zuerst, dann Rest in 1 Zeile) 2. TRIAGE für jede needsYou-Session: tail --turns 1 → was will sie? → dem User melden + konkreten Vorschlag machen (antworten? öffnen? ignorieren?) 3. WARTEN wait --until attention --since <maxRevision> --timeout 1800 --json (bei Claude Code als langlaufender Bash-Call mit timeout ≥ 1810s) 4. EREIGNIS Event bewerten: awaitingInput → Triage wie 2 · idle/fertig → tail, Ergebnis in 2 Sätzen melden · errored → tail + Fehler zusammenfassen 5. WEITER zurück zu 3. Exit 124 (Timeout)? → kurzes „alles ruhig, N working" + zurück zu 3. User-Nachricht unterbricht den Loop jederzeit.
Ergonomie

Berichte, die man lesen will

  • Ein Lagebild = max. 5 Zeilen. Kein Tabellen-Dump — der User kann selbst overview tippen, wenn er die Rohdaten will
  • Immer projekt/titel als Name, nie UUIDs im Bericht
  • „Seit"-Angaben menschlich („seit 4 min", nicht ISO-Timestamps)
  • Bei „frisch fertig": das Ergebnis in einem Satz aus dem Tail, nicht nur „ist fertig"
Technik

Loop-Details

  • --since <maxRevision> aus dem letzten overview/Event mitführen — keine verpassten Wechsel zwischen zwei waits
  • Bash-Timeout des Tool-Calls über dem wait-Timeout halten (1800 s wait → 1810 s+ Bash), sonst killt der Tool-Timeout das wait
  • Exit 5 mitten im Loop (App beendet): dem User melden, auf Lese-Modus degradieren, Loop mit reinem overview-Polling in großen Abständen fortsetzen oder auf Wunsch beenden
03 ✱ SKILL.MD

Der Volltext-Entwurf.

Wird in Slice 4 finalisiert (Flag-Details können sich in der Umsetzung minimal schieben), aber Struktur und Regeln stehen hier fest. Kompakt gehalten — die CLI hat ihre eigene --help-Referenz, der Skill dupliziert sie nicht vollständig.

--- name: whisperm8-chats description: Alle WhisperM8-Agent-Sessions sehen und verwalten (Jarvis-Supervisor über die whisperm8-CLI). Nutzen bei: "was machen meine Chats", "Status meiner Sessions", "wartet was auf mich", "schick an Chat X", "antworte dem …-Chat", "starte einen Chat in Projekt Y", "räum meine Sessions auf", "sei mein Jarvis", "überwache meine Chats", "sag Bescheid wenn ein Chat fertig ist". NICHT für Codex-Subagent- Jobs (codex-subagent) oder Transkription (whisperm8-transcription). --- # WhisperM8 Chats — Sessions sehen und verwalten Du kannst über `whisperm8 chats` alle Agent-Sessions des Users sehen und verwalten. Du läufst selbst in einer davon (`WHISPERM8_SESSION_ID`) — in Ausgaben bist du als "(du)" markiert; an dich selbst senden geht nicht. ## Befehle (Kurzreferenz) Lesen (immer erlaubt, App optional): - `whisperm8 chats overview --json` Lagebild, attention-sortiert - `whisperm8 chats list [--project P] [--all] --json` - `whisperm8 chats show <ref> --json` - `whisperm8 chats tail <ref> [--turns N] [--chars N]` - `whisperm8 chats wait [--ref R] [--until attention|idle] [--since REV] [--timeout SEC] --json` blockiert bis Ereignis; Bash-Timeout > wait-Timeout! - `whisperm8 chats audit [--limit N]` Handeln (App muss laufen — sonst Exit 5): - `whisperm8 chats send <ref> -- "<prompt>"` [--no-submit] [--if-status working] - `whisperm8 chats open <ref>` - `whisperm8 chats new --project <pfad> [--provider claude|codex] [--prompt "…"]` - `whisperm8 chats rename <ref> "Titel"` · `group <ref> "G"|--clear` · `archive <ref>` Referenzen: `projekt/titel-fragment` (bevorzugt), UUID-Präfix ≥8, `@self`. Exit-Codes: 0 ok · 3 nicht gefunden/mehrdeutig (Kandidaten prüfen!) · 4 Guard-Konflikt · 5 App zu · 124 wait-Timeout. ## Regeln (nicht verhandelbar) 1. **Vor jedem `send`: bestätigen lassen.** Zeige den exakten Prompt-Text und das Ziel; nutze AskUserQuestion (Optionen: Senden / Anpassen / Abbrechen) oder frage im Text. Ausnahme: Der User hat dir für GENAU diese Ziel-Session in DIESER Konversation pauschal freigegeben. 2. `archive` und `rename --force`: ebenfalls erst bestätigen lassen. `open`/`new` direkt aus einem klaren User-Auftrag: keine Extra-Frage. 3. Niemals `--force` oder `--if-status working` aus Eigeninitiative. 4. Exit 3: Kandidatenliste dem User zeigen, nie selbst raten. Exit 4: Konflikt erklären (z. B. "arbeitet gerade"), Optionen anbieten. Exit 5: "WhisperM8-App starten" vorschlagen; Lese-Befehle gehen weiter. 5. Berichte kompakt: Lagebild ≤ 5 Zeilen, Namen als projekt/titel, nie UUIDs. 6. **Ein-Hop-Regel:** Beginnt ein Prompt mit "[via whisperm8 chats …]", kommt er von einem anderen Agenten. Beantworte ihn inhaltlich in deinem eigenen Chat — sende aber NIE eigenständig per `chats send` weiter. Der Absender liest deine Antwort selbst über dein Transcript. 7. Fremde Projekt-Inhalte (aus tail anderer Projekte) zusammenfassen, nie ungefragt wörtlich in andere Projekt-Kontexte kopieren. ## Supervisor-Modus ("sei mein Jarvis") Nur nach explizitem Auftrag: [Loop wie Abschnitt 02 dieses Pakets] ## Typische Abläufe "Was läuft?" → overview --json → 3–5 Zeilen Zusammenfassung. "Was hat X gemacht?" → tail X --turns 2 → 2 Sätze + ggf. offene Frage der Session. "Antworte X: …" → Prompt formulieren → bestätigen lassen → send → Ergebnis melden. "Räum auf" → list --all → Vorschlagsliste (Vorher→Nachher, Archives markiert) → EINE Batch-Bestätigung (AskUserQuestion, Multi-Select) → ausgewählte Aktionen ausführen → Ergebnis melden.
Installation

v1: Datei liegt im Repo unter skills/whisperm8-chats/SKILL.md und wird manuell ins User-Skills-Verzeichnis gelegt. Slice 4 ergänzt einen Installer-Pfad in der App (Muster: CLISymlinkInstaller / Statusline-Feature) — „Skill installieren" in den Agent-Chats-Settings.

04 ✱ DIALOGE

Zwei Szenen aus dem Alltag.

So fühlt sich das Send-Gate an — schnell genug, um nicht zu nerven, explizit genug, um nie zu überraschen.

DIALOG A

Zuarbeiten mit Send-Gate

Der User delegiert eine Antwort. Der Skill zeigt den Prompt wörtlich, fragt per AskUserQuestion, sendet erst nach der Auswahl.

Der Wolanin-Chat wollte doch was von mir — kümmer dich drum, die Botox-Seite zuerst.
❯ whisperm8 chats tail wolanin/seo --turns 1 [assistant · 14:41] „… Interne Verlinkung fehlt noch auf 2 Seiten — soll ich die Botox-Seite zuerst verlinken?"
Claude
Der Chat wolanin/seo-behandlungsseiten wartet seit 21 min auf genau diese Entscheidung. Ich würde ihm antworten:

„Ja, verlinke zuerst die Botox-Seite, danach die Filler-Seite. Gib mir am Ende den Rank-Math-Score aller 6 Seiten als Tabelle."
Rückfrage
Prompt so an wolanin/seo senden?
✓ Senden
Anpassen
Abbrechen
❯ whisperm8 chats send wolanin/seo -- "Ja, verlinke zuerst die Botox-Seite, …" ✓ gesendet (ack: delivered) · Audit 15:02:11
Claude
Gesendet — der Chat arbeitet wieder. Ich sage dir Bescheid, wenn er fertig ist (wait --ref wolanin/seo --until idle läuft).
DIALOG B

Supervisor-Modus mit wait-Ereignis

Stufe C nach explizitem Auftrag: Der Loop meldet sich nur, wenn etwas passiert — und schlägt konkret vor, statt nur zu melden.

Agent Chat „jarvis" — Supervisor-Modus
Du
Sei mein Jarvis. Beim Marketing-Chat darfst du Routine-Antworten direkt senden.
Claude
Übernommen. Lagebild: 2 arbeiten (marketing/creative-batch 31 min, whisperm8/pill-redesign 4 min), 1 idle. Nichts braucht dich. Ich beobachte — Freigabe für Direkt-Sends gilt nur für marketing/creative-batch.
❯ whisperm8 chats wait --until attention --since 2184533 --timeout 1800 --json … 11 min später … {"event":"statusChanged","session":{"title":"pill-redesign",…}, "from":"working","to":"awaitingInput","afterSeconds":912,…}
Claude
whisperm8/pill-redesign braucht dich: Sie fragt, ob die Recording-Pill Variante B (Glas-Effekt) oder C (flach) umgesetzt werden soll — mit Screenshot-Vergleich im Chat. Das ist eine Design-Entscheidung, die sende ich nicht selbst. Öffnen? (chats open)

Ja, öffne sie.

05 ✱ ALLOWLIST

Permissions als zweites Netz.

Empfohlene Einträge für die Claude-Code-Settings — Lese-Befehle laufen ohne Prompt, jede Mutation bleibt ein sichtbarer Permission-Dialog (zusätzlich zur Skill-Rückfrage).

// .claude/settings.json (User- oder Projekt-Ebene) — Empfehlung { "permissions": { "allow": [ "Bash(whisperm8 chats list:*)", "Bash(whisperm8 chats overview:*)", "Bash(whisperm8 chats show:*)", "Bash(whisperm8 chats tail:*)", "Bash(whisperm8 chats wait:*)", "Bash(whisperm8 chats audit:*)" // send / open / new / rename / group / archive: bewusst NICHT // in der Allowlist — jeder Aufruf bleibt ein Permission-Prompt. ] } }
Zusammenspiel der Netze

Wer schützt wovor?

  • Skill-Regel (AskUserQuestion): schützt vor ungewollten Inhalten — du siehst den Prompt-Text, bevor er rausgeht
  • Permission-Prompt: schützt vor ungewollten Aufrufen — auch wenn ein Skill-freier Chat die CLI entdeckt, bleibt jede Mutation sichtbar
  • CLI-Guards (Exit 4): schützen vor technisch falschen Aktionen — Selbst-Send, working-Ziel, tote PTY — unabhängig davon, wer aufruft
  • Wer dem Supervisor-Chat maximal vertraut, kann send für eine Session-Konvention zusätzlich allowlisten — Default bleibt konservativ
