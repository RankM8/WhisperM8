PAKET 03 ✱ SYSTEM ✱ v1

Ein Binary, zwei Rollen, ein Socket.

CLI und App sind derselbe Prozess-Code mit verschiedenen Einstiegen. Dieses Paket legt fest, wie sie miteinander reden: der AgentControlServer in der App, das NDJSON-Protokoll über den Unix Domain Socket, die Identitäts-Injektion beim PTY-Spawn — und die Gotchas, an denen so etwas sonst scheitert.

00 ✱ ÜBERSICHT

Entscheidung: BSD-UDS statt NWListener.

Das Repo hat mit dem ClaudeGPTMixRouter bereits einen NWListener-Loopback-HTTP-Server als Vorbild. Trotzdem fällt die Wahl für den Control-Socket auf einen klassischen BSD-Unix-Domain-Socket — aus drei Gründen.

Warum UDS gewinnt

Sicherheit ist hier Dateisystem-Sache

  • Zugriffskontrolle über Datei-Permissions (0700-Ordner + 0600-Socket) statt „jeder lokale Prozess darf auf den TCP-Port" + Token-Tanz
  • getpeereid() liefert die EUID des Gegenübers — Cross-User-Zugriff ist damit hart ausgeschlossen; NWListener exponiert den FD dafür nicht
  • Kein Port-Management: kein „Port belegt", kein Firewall-Prompt, kein Erraten des Ports durch die CLI
Wo NWListener bleibt

MixRouter bleibt, wie er ist

  • Der MixRouter braucht HTTP+SSE-Streaming für Claude-Backends — dafür ist NWListener richtig; er wird nicht angefasst
  • Der Control-Socket ist Request/Response mit Kleinst-Payloads — dafür reichen ~200 Zeilen BSD-Socket + DispatchSourceRead
  • SwiftNIO wäre für einen lokalen Ein-Client-RPC Overkill (GPT-Review, bestätigt)

Startpunkt des Servers

Der AgentControlServer startet beim App-Launch (analog AgentScanCoordinator), räumt beim willTerminate auf und ist per Default aktiv. Kill-Switch analog zum Event-Watcher: defaults write com.whisperm8.app agentControlServerEnabled -bool NO.

02 ✱ PROTOKOLL

NDJSON über einen abgesicherten Socket.

Eine Zeile = ein JSON-Objekt (Newlines in Strings sind JSON-escaped — Nachrichtengrenzen sind damit sicher), max. 1 MiB pro Zeile, eine Response pro Request, Verbindung pro CLI-Aufruf. Debugbar mit nc -U.

// Request (CLI → App) — eine Zeile { "protocolVersion": 1, "requestID": "f4e1a7c2-…", "actor": { "sessionID": "9c11d3e2-…", "token": "kZx…" }, // beides null wenn außerhalb einer WhisperM8-PTY "method": "session.send", "params": { "targetSessionID": "6f2b41a0-…", "prompt": "Ja, verlinke…", "ifStatus": ["awaitingInput","idle"], "submit": true } } // Response (App → CLI) — eine Zeile { "protocolVersion": 1, "requestID": "f4e1a7c2-…", "ok": true, "result": { "ack": "delivered", "target": { "id": "6f2b…", "title": "seo-behandlungsseiten" } } } // Fehler-Response — code mappt auf Exit-Codes der CLI { "protocolVersion": 1, "requestID": "…", "ok": false, "error": { "code": "conflict", // notFound | conflict | selfSend | noPty | internal "message": "Ziel ist working (seit 31 min)", "detail": { "currentStatus": "working" } } }
ping
Handshake: {appVersion, protocolVersion, pid}. CLI prüft protocolVersion — Mismatch → klarer Fehler statt stiller Inkompatibilität (Update-Fall: alter CLI-Prozess trifft neue App).
sessions.live
Autoritative Runtime-Daten für die Lese-Befehle: [{sessionID, runtimeStatus, since, isAttachedPTY, canSend}] aus AgentSessionRuntimeStatusStore + AgentTerminalRegistry. list/overview/show mergen das über den Disk-Stand (live: true).
session.send
Atomare Send-Pipeline (Abschnitt 04). Ack-Stufen: delivered (gepastet), duplicate (requestID im 60-s-Cache).
session.open
WindowRequestCenter.requestSessionFocus + App aktivieren (NSApp.activate).
session.new
upsertProject → createSession(shouldLaunchOnOpen: true) → Fokus-Request. Antwort nach Persistenz-Enqueue mit Session-ID.
workspace.rename / .group / .archive
Guards prüfen (hoisted), dann AgentSessionStore-Mutation. Vorher/Nachher in die Antwort + Audit.
Dateisystem-Härtung

Socket-Lebenszyklus (GPT-Review übernommen)

  • Ordner …/WhisperM8/control/ mit 0700; Socket control.sock nach bind auf 0600 — der restriktive Elternordner schließt das bind→chmod-Fenster
  • sun_path-Limit (104 Bytes): UTF-8-Bytelänge des Pfads vor bind prüfen; zu lang → Fallback /private/tmp/whisperm8-<uid>/control.sock (0700-Ordner), Pfad in Discovery-Datei control/socket-path publiziert; CLI liest erst die Discovery-Datei
  • Stale-Socket: Server hält flock auf control.lock. Bei EADDRINUSE: erst connect+ping versuchen; nur wenn Connect scheitert UND lstat „Socket + eigene UID" bestätigt → unlink + einmal neu binden. Nie blind löschen
  • Shutdown: nur unlinken, wenn dev+ino noch dem eigenen bind entsprechen
  • Pro Verbindung getpeereid(): Peer-EUID ≠ App-EUID → sofort schließen
Server-Verhalten

Klein, seriell, off-main

  • Accept + Read + JSON-Decode auf einer Utility-Queue — nie auf dem MainActor lesen/parsen
  • Validierte Commands hoppen per await MainActor.run in die Handler; Ergebnis zurück auf die Queue, Response schreiben, Verbindung schließen
  • Limits: max. 4 gleichzeitige Verbindungen, 5 s Read-Timeout, 1 MiB Zeile — alles darüber: Fehler-Response + close
  • Requests seriell ausführen (eine Handler-Queue): Mutationen sind selten und klein; Serialität ist das billigste Konsistenzmodell
03 ✱ IDENTITÄT

Wer ruft hier eigentlich an?

Selbst-Send-Schutz und Audit brauchen eine Antwort auf „welche Session führt diesen Bash-Befehl aus?". Die App weiß es, weil sie die PTYs selbst spawnt.

BEIM PTY-SPAWN

AgentTerminalController.start() injiziert zwei Variablen

Direkt vor startProcess (Views/AgentTerminalView.swift:758 — die einzige Stelle, wo Env-Array und sessionID garantiert zusammen vorliegen): WHISPERM8_SESSION_ID=<uuid> + WHISPERM8_SESSION_TOKEN=<32 zufällige Bytes, base64>. Token wird im AgentControlServer registriert (in-memory, pro Controller-Lauf neu).

Kind-Prozesse (claude → bash → whisperm8) erben die Env
BEIM CLI-AUFRUF

CLI liest die Env und legt sie in den Request

actor.sessionID steuert @self, „(du)"-Markierung und Selbst-Send-Schutz. actor.token beweist der App: der Aufruf kommt wirklich aus dieser PTY — die nackte Session-ID wäre spoofbar (jeder Prozess des Users kann sie raten).

Server: Token bekannt und passend zur Session-ID?
BEWERTUNG

Verifiziert / Unverifiziert / Extern

Token gültig → Actor verifiziert (Audit: projekt/titel). Session-ID ohne/mit falschem Token → Aufruf läuft, aber als actor: "unverified" im Audit; Selbst-Send-Schutz greift trotzdem über die ID. Ganz ohne Env (User-Terminal) → actor: "external". Ehrliche Einordnung: Das ist Rechenschaft + Versehens-Schutz, keine Security-Boundary gegen einen absichtlich bösartigen Prozess desselben Users — der könnte auch direkt die App bedienen.

Audit-Log

chats-audit.jsonl im App-Support-Ordner, geschrieben nur von der App (eine Schreibstelle, kein Locking-Problem): {at, actor, verified, method, target, outcome, promptChars, promptHead80}. Rotation bei 5 MB → .1-Sidecar. Kein Prompt-Volltext — Privacy-Default; das Transcript der Ziel-Session hat den Volltext ohnehin.

04 ✱ SEND-PIPELINE

Der kritische Abschnitt, im Detail.

Das TOCTOU-Problem aus dem GPT-Review, gelöst: Alle Prüfungen und der Paste passieren in einem synchronen MainActor-Block — zwischen Check und Injektion liegt kein await.

1 · EINGANG (Server-Queue)

Decode + Idempotenz

requestID im 60-s-Cache? → sofort ack: "duplicate", kein zweiter Paste. Sonst: Prompt-Länge validieren (≤ 100 kB), dann MainActor-Hop.

await MainActor.run { … alles Folgende synchron … }
2 · ATOMARE PRÜFUNG (MainActor, synchron)

Fünf Guards, eine Reihenfolge

① Session existiert + nicht archiviert (sonst notFound) → ② Ziel ≠ Actor-Session (sonst selfSend, kein Override) → ③ AgentTerminalRegistry.controller(for:) existiert + isRunning (sonst noPty) → ④ Runtime-Status ∈ ifStatus aus dem Request (sonst conflict + aktueller Status) → ⑤ Prompt nicht leer.

3 · PASTE (MainActor, synchron)

Bracketed Paste als ein Block

controller.sendPrompt(text, submit:) — neu in Slice 2: ESC[200~ + Text + ESC[201~ via terminal.send(txt:). Der Wrap verhindert, dass die Claude/Codex-TUI jede Newline als Submit deutet. Requeue der requestID in den Idempotenz-Cache passiert vor dem Paste.

submit == true → nach 80 ms (Task.sleep, außerhalb des kritischen Blocks):
4 · SUBMIT + ANTWORT

CR, Audit, delivered

Ein 0x0d an die PTY (Delay lässt die TUI den Paste verarbeiten — Muster analog zur bestehenden 1,0-s-Grace im AgentPromptRoutingService). Audit-Zeile schreiben, ack: "delivered" zurück. Der Submit ist best-effort nach erfolgreichem Paste — ein Fehler hier wird geloggt, ändert aber die Antwort nicht mehr (dokumentiert).

Warum nicht AgentPromptRoutingService?

Bewusste Trennung vom Diktat-Pfad

  • Der bestehende route(text:toLocalSessionID:) ist für Diktat gebaut: Fokus + Session-Start + Retries + ohne Submit — genau richtig für „User diktiert in seinen aktiven Chat"
  • chats send braucht das Gegenteil: kein Fokus-Klau (der User arbeitet vielleicht woanders), kein Auto-Start einer toten Session, dafür atomare Guards + Submit
  • Beide teilen sich künftig sendPrompt(_:submit:) am Controller als gemeinsamen Baustein — mehr nicht
05 ✱ GOTCHAS

Woran es sonst scheitert.

STORE-LOCK-REGEL
Mutation-Closures im AgentWorkspaceStore laufen unter dem prozessweiten Lock — keine Subprozesse, kein I/O darin. Alle Guard-Prüfungen vor dem mutate-Aufruf hoisten (bestehende Projektregel, gilt für alle workspace.*-Handler).
NIE loadWorkspace() ZUM REFRESHEN
App-seitig observieren die Handler AgentWorkspaceUIModel; CLI-seitig liest ChatsWorkspaceReader nur die Repository-Disk-Funktion. loadUIState() ist tabu im CLI (persistiert als Nebenwirkung).
CLI-PROZESS = APP-BINARY
Der CLI-Pfad darf keine App-Singletons anfassen, die Seiteneffekte starten (AgentScanCoordinator, Stores, Watcher). Nur die puren/statischen Bausteine + Repository. Review-Checkliste in Paket 05.
SANDBOX-KINDER
Codex-Subagents laufen teils sandboxed (--sandbox read-only) — deren whisperm8 chats-Lese-Befehle funktionieren (Disk-Reads), aber der Socket-connect kann geblockt sein → sauberer Exit 5 mit Hinweis, kein Hänger (Connect-Timeout 2 s).
LOGIN-SHELL-ENV
Falls App-Handler je Subprozesse spawnen (z. B. künftiges new mit Worktree): immer LoginShellEnvironment, nie ProcessInfo-Env (Projektregel). In v1 spawnen die Handler nichts.
MAKE DEV IN EIGENER SESSION
Slice-2+-Tests brauchen die neue App. Wer als Agent-Chat IN WhisperM8 läuft, darf make dev/make kill nicht ausführen (killt die eigene Session, CLAUDE.md-Regel) — der User startet die App selbst neu. Für die Umsetzung heißt das: Socket-Tests als swift test mit In-Process-Server, nicht gegen die echte App.
PERFORMANCE-BUDGETS
Neue Signpost-Kategorie nicht nötig; aber sessions.live-Handler und Send-Pipeline bleiben unter dem bestehenden perf.store-Mutations-Budget (30 ms) — nichts Teures auf dem MainActor.
DISCOVERY-DATEI
CLI findet den Socket über control/socket-path (ein Pfad, eine Zeile). Fehlt die Datei → App lief nie / Feature aus → Exit 5 mit passender Meldung. Nicht raten, nicht mehrere Pfade probieren.
