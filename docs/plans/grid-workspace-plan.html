Grid-Workspace: feste Slots, echtes Drag & Drop, 2–9 Panes

Das Grid wird vom Tab-Ausschnitt zum kuratierten Workspace: Du bestimmst, welcher Chat in welchem Slot liegt, sortierst per Drag um, ersetzt gezielt per Reinschieben — und keine Automatik verdrängt mehr irgendetwas. Produktgrundsatz: Ein Klick navigiert, ein Drop kuratiert. Kapazität neu: bis 3×3 = 9 Panes.

01 — INTERAKTIVER MOCKUPZieh es dir zurecht

Alles echt draggable: Chips oben (deine Tabs) auf eine Pane ziehen = ersetzen, auf einen leeren Slot = einfügen. Pane-Header auf eine andere Pane ziehen = tauschen, auf einen leeren Slot = verschieben. Klick auf eine Pane = Fokus. ⊖ leert den Slot (Tab bleibt). Rechts die Kapazität — bis 3×3.

Chip → Pane = ersetzen Header → Pane = tauschen → leerer Slot = einfügen volles Grid + Drop = wächst
Tabs
Slots bleiben stabil — nichts rückt nach, keine Verdrängung
＋ Hier ablegen — Grid erweitern
Regeln (aus der UX-Konsultation, verbindlich für die Umsetzung)

„Ein Klick navigiert, ein Drop kuratiert"

  • Feste Slots mit expliziter Belegung; leere Slots bleiben sichtbar als Drop-Fläche — Positionen sind stabil, nichts kollabiert
  • Sidebar-/Tab-Klick auf einen Chat außerhalb des Workspace ändert das Grid NIE: Einzelansicht öffnet sich, „Zurück zum Workspace" stellt exakt das Grid wieder her. Die Verdrängungs-Automatik (bringSelectionIntoGrid) wird ersatzlos entfernt
  • Ersetzen nimmt den alten Chat nur aus dem Workspace — Tab und Prozess laufen weiter
  • Auto-Wachsen — ein Drop blockiert nie: Ist das Grid voll, erscheint beim Hineinziehen eine „Grid erweitern"-Zone; der Drop wächst zur nächsten Stufe (2→3→4→6→9), der neue Chat landet im ersten neuen Slot. Geschrumpft wird nie automatisch (stabile Positionen!) — verkleinern nur über den Picker
  • Kontextmenü „Im Workspace platzieren → Slot N" als präziser Weg ohne Drag (auch für Bulk)
  • Drop-Feedback benennt die Aktion konkret: „X durch Y ersetzen", „Tauschen", Akzent-Kontur auf leeren Slots; unbeteiligte Panes dimmen
  • Maximize/Minimize (⤢/Grid-Button) und verschiebbare Trennlinien bleiben wie gebaut; Trennlinien sind KEINE Drop-Ziele (Resize und Einfügen nie auf derselben Fläche)

02 — KAPAZITÄTMehr als 4? Ja — bis 3×3.

Das bisherige 4er-Limit war eine konservative Setzung (TUI-Lesbarkeit + ungemessene Performance), keine technische Grenze. Mit dem Slot-Modell und dem Performance-Paket unten fällt sie. Natürliche Grenze

Mindest-Pane statt Hardcap

  • Pro Pane gelten ~240 pt Mindestbreite (GridSplitResolver.minPane) — auf kleinen Fenstern blendet der Kapazitäts-Picker Stufen aus, die nicht passen, statt Matchbox-Terminals zu rendern
  • 8 (4×2) lassen wir bewusst weg: 4 Spalten unterschreiten auf üblichen Fenstern die Lesbarkeit; 3×3 deckt „viele auf einmal" ab. Nachrüstbar, falls du es brauchst
Trennlinien bei 3 Spalten/Zeilen

Splits werden Arrays

  • Heute: EIN Verhältnis pro Achse. Bei 3 Spalten werden daraus zwei Divider pro Achse → GridSplitResolver wird auf Fraction-Arrays verallgemeinert (pure Logik + Tests, gleiche Clamping-Regeln)
  • Doppelklick auf eine Linie = diese Achse zurück auf Gleichverteilung

04 — TECHNIKWie es gebaut wird (Machbarkeit verifiziert)

Datenmodell

gridSlots: [UUID?] statt Mitglieder-Liste

  • Neuer JSON-Key, Länge 2–9; nil = leerer Slot. gridSessionIDs wird nur noch als Legacy gelesen (Migration in Bildschirm-Reihenfolge)
  • Manueller Decoder mit decodeIfPresent (etabliertes Muster gegen den dokumentierten Tab-State-Totalverlust)
  • normalizedWindows: geschlossene Tabs werden am Index zu nil — nicht kompaktiert, Positionen bleiben stabil
Drag & Drop

DropDelegate-Muster + Slot-Herkunft

  • GridPaneDropDelegate nach dem bewährten TabReorderDropDelegate-Muster (liefert Zonen + Move-Semantik); pure GridDropZoneResolver-Logik, unit-getestet
  • DraggableSession bekommt optionales sourceGridSlotIndex → eindeutige Swap- vs. Replace-Entscheidung; alte Payloads bleiben decodierbar
  • Drag-Quelle: nur der Pane-Header (Terminal-Selektion bleibt ungestört); Doppelklick-Maximize bleibt
SwiftTerm

Tausch ohne Prozessverlust

  • PTY/Scrollback hängen an der sessionID-Registry, nicht am Slot — Swap = zwei Re-Attaches, kein Neustart (verifiziert)
  • Härtung: Fokus nach Slot-Mutation explizit neu setzen (selectedSessionID ändert sich beim Tausch nicht) + Frames atomar in einer CATransaction committen (kein Flackern)
Was entfällt

Aufräumen

  • bringSelectionIntoGrid + Verdrängungs-Automatik: ersatzlos raus
  • Ganzflächiges Grid-dropDestination → ersetzt durch per-Slot-Ziele
  • Degenerierte-Mitgliedschaft-Heuristik („≤1 → Default") → leere Slots sind jetzt legal
