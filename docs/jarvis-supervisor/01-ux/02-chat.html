PAKET 02 ✱ CHAT ✱ SLICE 1 + 2

Ein Chat, der weiß, woran du arbeitest.

Der Supervisor-Chat sitzt über dem Board. Mit @-Mentions holst du gezielt Sessions in den Kontext — sie bekommen das volle Digest-Budget, alles andere bleibt Metadaten. Genau ein Codex-Aufruf pro Turn, und die UI ist ehrlich über die Latenz.

Enginecodex exec · seriell · 45–90 s Timeout
Purer KernJarvisMentionResolver
00 ✱ ÜBERSICHT

Was der Chat ist — und was nicht.

Kein Tool-Loop, kein Auto-Pilot: Der Chat beantwortet Fragen über den Workspace, entwirft Folgeprompts und schlägt Aufräumaktionen vor. Alles Mutierende landet als Karte unter „Aktionen".

Kann

Typische Turns

  • „Was ist liegengeblieben?" — Zusammenfassung über alle Attention-Sessions
  • „Vergleiche @a mit @b" — Cross-Projekt-Analyse mit vollen Digests der Erwähnten
  • „Entwirf den Folgeprompt für @x" — erzeugt JarvisPromptDraft unter Aktionen
  • „Räum den Workspace auf" — erzeugt Rename/Group/Archive-Vorschläge als Karten
Kann nicht (v1)

Bewusste Grenzen

  • Keine Laufzeit-Tools — Kontext wird vorab injiziert; Jarvis kann nicht „nachschlagen"
  • Keine Mutation direkt aus dem Chat — immer über die Action-Queue mit Apply
  • Kein Streaming — codex exec liefert erst am Ende; die Thinking-UI überbrückt
  • Keine AgentView-Subsessions — unsupportedAgentView, klar kommuniziert
Warum kein Tool-Loop

codex exec ist ein One-Shot-Prozess. Ein Read-only-Tool-Loop hieße mehrere Codex-Aufrufe pro Turn — Minuten Latenz, Kontext jedes Mal neu bezahlt. Der v1-Schnitt: alles Relevante steht schon im Prompt, gesteuert über @-Mentions. Echter Tool-Loop kommt in v2 als MCP.

01 ✱ KONTEXTBUDGET

Wer erwähnt wird, bekommt Budget.

Ohne explizite Referenzen müsste Jarvis raten, wofür er Kontext ausgibt — und du zahlst Digests für 20 Sessions, von denen dich zwei interessieren.

Volles Budget · ~12k
Digest + Tail für: @-erwähnte Sessions, die aktuell selektierte Session, Sessions mit frischem ungelesenem Statuswechsel (bei Report-/Cleanup-Turns). Ein erwähntes Projekt zieht seine aktiven Sessions mit kleinem Budget nach.
Metadaten · ~200
Alle übrigen Sessions: Snapshot-Zeile (Titel, Projekt, Status, Dauer) + Tier-1-Einzeiler. Genug für „was läuft"-Fragen, zu wenig für Inhaltsanalyse — das ist Absicht.
Gesamt · ~60k
Hartes Prompt-Budget. Bei Überschreitung werden Digests in Attention-Reihenfolge gekürzt, und der Prompt vermerkt explizit, was gekürzt wurde — Jarvis behauptet nie, etwas gelesen zu haben, das nicht drin war.
Resolver
JarvisMentionResolver ist pur und unit-getestet: Text + Snapshot → aufgelöste Session-IDs. Fuzzy-Match auf Titel und Projektname; mehrdeutige Treffer zeigt der Picker, er rät nicht.
02 ✱ MOCKUPS

Picker, Denken, Scheitern.

Die drei Momente, die über das Chat-Gefühl entscheiden: Kontext wählen, warten können, Fehler aushalten.

Interactions

Picker-Verhalten

  • „@" tippen öffnet den Picker; weitertippen filtert fuzzy über Titel + Projektname
  • ↑/↓ + Enter wählt; Esc schließt und lässt das „@" als Text stehen
  • Match-Highlights im Treffer (blau) zeigen, warum ein Eintrag matcht
  • Projekt wählen = Chip @projekt; im Turn bekommen dessen aktive Sessions kleines Budget
  • Chips sind pro Turn — nach dem Senden bleiben sie sichtbar, bis man sie entfernt (Folge-Fragen zur selben Session)
States

Sortierung im Picker

  • Erst Attention-Kategorie (wer dich braucht, steht oben), dann Fuzzy-Score
  • Session-Einträge zeigen Statuspunkt + Projekt; Projekt-Einträge die Session-Anzahl
  • Max. 6 Treffer sichtbar, Rest scrollt — der Picker ist ein Schnellzugriff, kein Browser
  • Archivierte Sessions erscheinen nicht (Snapshot enthält sie nicht)

States

Thinking-Mechanik

  • Schritt 1 + 2 sind echte App-Arbeit (Snapshot, Digests) — der Fortschritt ist nicht simuliert
  • Schritt 3 („Codex denkt") zeigt verstrichene Zeit ab 10 s: Codex denkt · 22 s
  • Abbrechen (✕) killt den Codex-Prozess; der Turn landet als abgebrochen im Verlauf
  • Timeout 45 s (Chat) / 90 s (Cleanup/Briefing) → Fehler-State, kein ewiger Spinner
Interactions

Während des Turns

  • Composer gesperrt, aber Segmente frei — Board und Aktionen bleiben voll bedienbar
  • Neue Tier-1-Reports erscheinen währenddessen normal (unabhängige Schicht)
  • Folge-Turns landen in der seriellen Queue; die UI zeigt „1 wartend"
  • Antworten verweisen auf erzeugte Drafts/Actions per Inline-Button statt Inhalte zu duplizieren

Regel

Snapshot-Fragen kosten nichts

  • „Wer wartet?", „Was läuft?", „Wie lange läuft X schon?" — beantwortet das Attention-Modell direkt, gerendert als normale Jarvis-Nachricht mit „SOFORT"-Kennung
  • Erkennung über die Quick-Reply-Chips (klick = garantiert LLM-frei) — Freitext geht im Zweifel an Codex
  • „Briefing ✦" ist als einziger Chip ein Codex-Turn (gebatchte Tier-2-Zusammenfassung) — das ✦ markiert LLM-Kosten
Failure

Fehler-Verhalten

  • Codex fehlt/abgemeldet: Header-Status CODEX ERROR, Composer deaktiviert, Setup-Hinweis; alles andere läuft
  • Kaputtes JSON (auch nach Fence-Stripping): Raw-Output als failed-Event einsehbar, keine Actions erzeugt
  • Timeout: klare Meldung mit „Erneut senden" — der Prompt geht nicht verloren
03 ✱ TURN-ABLAUF

Ein Turn, fünf Schritte.

1 · Resolve
JarvisMentionResolver löst Chips + Inline-@ gegen den Snapshot auf. Mehrdeutig → Picker fragt nach, der Turn startet erst danach.
2 · Collect
Snapshot (MainActor) → Sendable-Kopie; Digests für Budget-Sessions off-main aus dem LRU-Cache oder frisch (mtime/size-Key).
3 · Build
Prompt Builder: Systemprompt + Snapshot kompakt + Digests + letzte N Chat-Messages + Action-Schemas + Auftrag. Budget-Kürzung in Attention-Reihenfolge, Kürzungen im Prompt vermerkt.
4 · Execute
codex exec --sandbox read-only, Prompt via stdin, Antwort via --output-last-message. Serielle Queue, Timeout 45/90 s, Prozess-Kill bei Abbruch.
5 · Apply
Fence-toleranter Strict-JSON-Parser. IDs vergibt die App. Message → Chat, Action-Drafts → validiert → Queue, Prompt-Drafts → Aktionen-Segment. Ungültiges wird geloggt und verworfen.

Parser-Leitplanke

Draft-Typen referenzieren Sessions über Titel/Kontext — der Parser mappt gegen den Snapshot und verwirft, was nicht eindeutig auflösbar ist. Codex erfindet keine UUIDs, keine Timestamps, keine Stati.

05 ✱ EDGE-CASES

Chat-spezifische Kanten.

Mention auf tote Session
Session wurde zwischen Chip und Senden archiviert/gelöscht → Resolver schlägt beim Turn-Start fehl, Chip färbt rot, Turn startet nicht. Kein halber Kontext.
Digest zu groß
Drei @-Mentions à 12k sprengen das 60k-Gesamtbudget → Kürzung in Attention-Reihenfolge; die Antwort beginnt mit dem Hinweis, welche Session nur teilweise gelesen wurde.
Antwort behauptet Mutation
Systemprompt verbietet es; falls Codex trotzdem „ich habe umbenannt" schreibt, ohne Action-Draft: Message wird angezeigt, aber der Parser hängt einen Systemhinweis an („keine Aktion erzeugt").
App-Neustart mid-Turn
Laufender Codex-Prozess stirbt mit der App. Beim Start: offene Turns als cancelled markieren — nie ein ewiger THINKING-Header aus persistiertem State.
200-Message-Prune
Chat-Verlauf ist Arbeitskontext, kein Archiv. Prune schneidet älteste zuerst; der Prompt Builder nutzt ohnehin nur die letzten N Messages.
