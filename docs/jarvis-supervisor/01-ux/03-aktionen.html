PAKET 03 ✱ AKTIONEN ✱ SLICE 3–4

Vorschlagen darf Jarvis. Anfassen darfst du.

Der Review-Layer: Jede Mutation ist eine Karte mit Vorher/Nachher und Begründung. Apply führt aus, Skip verwirft, Undo dreht zurück. Und der Höhepunkt des Loops: der Prompt-Draft, der per Bracketed Paste in die laufende PTY geht.

00 ✱ ÜBERSICHT

Review-first, konkret.

Aus dem früheren Tool-Registry-Konzept bleibt der mutierende Teil: JarvisActionRegistry mit MCP-kompatibel benannten Schemas — v2 kann sie 1:1 als MCP-Tools exportieren.

Input

Woher Actions kommen

  • Brain-Turns — Chat-Antworten enthalten actions[]-Drafts (ohne IDs)
  • „Workspace aufräumen"-Button — ein Cleanup-Turn über den ganzen Snapshot
  • „Folgeprompt entwerfen" auf Report-Karten — erzeugt Prompt-Drafts
  • Parser validiert Drafts gegen den Snapshot, mappt Titel → Session-ID, verwirft Unauflösbares
Output

Wohin Mutationen gehen

  • rename / group / archive → AgentSessionStore (nie am Workspace-Store vorbei)
  • Tab öffnen / Session selektieren → Host-Callbacks aus AgentChatsView
  • PTY-Send → AgentTerminalController.sendPrompt (Bracketed Paste)
  • Validierung vor dem mutate-Aufruf — nichts Teures unter dem Store-Lock
Undo als Vertrauens-Hebel

Rename/Group/Archive speichern den before-Wert — ein Klick „Rückgängig" auf angewendeten Karten stellt ihn wieder her. Das senkt die Hemmschwelle beim Reviewen deutlich und ist der Pfad, über den v2 Auto-Apply für reversible Actions anbieten kann.

01 ✱ ACTION-KINDS

Fünf Kinds, klare Regeln.

rename_session
Batch-fähig. Schutz: bei manuell gesetztem Titel (titleIsAutoGenerated != true) nur erlaubt, wenn die Action explizit als nutzer-angefragte Cleanup-Aktion markiert ist. Undo: alter Titel.
set_session_group
Batch-fähig. Gruppiert Sessions thematisch (z. B. Sprint-Gruppen). Undo: alte Gruppe (auch nil).
archive_session
Nie im Batch. Abgelehnt, wenn Session working ist oder ein PTY läuft. Undo: Status zurück auf den vorherigen Wert.
create_prompt_draft
Erzeugt einen JarvisPromptDraft — selbst keine Mutation am Workspace, darum ohne Apply-Schritt direkt sichtbar. Wird erst durch Copy/Send „benutzt".
send_prompt_draft_to_running_pty
Nie im Batch, kein Undo. Nur bei laufendem AgentTerminalController; sonst ist der Button deaktiviert und Copy/Tab-öffnen übernehmen. Höchste Review-Stufe: expliziter Klick pro Send.

Verwerfen
States

Karten-Lebenszyklus

  • pending → applied / skipped / failed / reverted
  • Applied-Karten bleiben mit Undo-Button sichtbar, bis sie aus dem Prune fallen (max. 100)
  • Failed zeigt die konkrete Ablehnung — Drift ist eine Information, kein stiller Fehler
  • Batch-Apply wendet nur die sicheren Karten der Gruppe an; jede validiert einzeln
Apply-Validierung

Beim Klick, nicht nur beim Vorschlag

  • Session existiert noch? Status kompatibel? (Archive nicht bei working/PTY)
  • Drift-Check: ist der before-Wert noch aktuell? Wenn nicht → failed mit Meldung
  • Validierung + Git/Subprozess-freie Vorarbeit vor dem Store-mutate (Lock-Gotcha)

Bracketed Paste — der technische Kern des Send

Warum roh senden kaputt wäre

  • Mehrzeiliger Prompt roh an terminal.send(txt:) → jede Newline submitted sofort in der Claude/Codex-TUI. Der Prompt kommt als fünf halbe Nachrichten an.
  • Fix: Text in ESC[200~ … ESC[201~ wrappen — die TUI behandelt ihn als einen Paste-Block
  • submit == true: Return erst nach dem Paste-Block, mit ~50–100 ms Delay, damit die TUI den Paste verarbeitet hat
  • V1 sendet ausschließlich nach explizitem Button-Klick — kein Auto-Send, auch nicht als Setting
  • Draft-Status wandert: draft → copied / sent / failed; „sent" zeigt Uhrzeit
03 ✱ APPLY-PIPELINE

Vom Klick zur Mutation.

1 · Revalidate
Snapshot frisch ziehen: Session existiert, Status kompatibel, before-Wert stimmt noch. Alles außerhalb des Store-Locks.
2 · Mutate
Kürzest möglicher AgentSessionStore-Aufruf: renameSession / setSessionGroup / updateSession {{ status = .archived }}. Keine Subprozesse, kein I/O im Closure.
3 · Record
Action-Status auf applied, before-Wert bleibt für Undo gespeichert. UI-nahe Effekte (Tab selektieren) über Host-Callback danach.
Undo
Gleiche Pipeline rückwärts mit vertauschtem before/after; Status → reverted. Undo revalidiert genauso — wer den Titel inzwischen manuell geändert hat, bekommt failed statt Überschreiben.

05 ✱ EDGE-CASES

Die Kanten des Review-Layers.

Drift zwischen Proposal & Apply
Nutzer benennt manuell um, während die Karte offen ist → Drift-Check schlägt an, Karte wird failed mit Erklärung. Nie stilles Überschreiben.
PTY stirbt zwischen Klick & Send
Send prüft den Controller im Moment des Sendens. Weg? → Draft-Status failed mit „Session nicht mehr aktiv", Copy/Tab-öffnen bleiben.
Session mid-Turn beim Send
Ziel-Session ist working → Send-Button zeigt „Senden nach Turn-Ende" als Tooltip und bleibt aus. Ein Prompt mitten in einen laufenden Turn zu pasten erzeugt Chaos.
Doppel-Apply
Zwei Fenster, dieselbe Karte: der zweite Apply trifft auf Status applied → No-op mit kurzem Hinweis. Action-Status ist Teil des globalen Stores, nicht fensterlokal.
Codex schlägt Unbekanntes vor
Unbekannter kind oder nicht auflösbare Session → Parser verwirft und loggt. Die Queue zeigt nur, was die App auch ausführen kann.
