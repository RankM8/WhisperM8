PAKET 01 ✱ ÜBERSICHT ✱ SLICE 1–2

Das Board, das dir sagt, wer dich braucht.

Standard-Segment des Jarvis-Inspectors. Alles hier entsteht ohne LLM: Runtime-Status, Attention-Sortierung, Tier-1-Reports aus Transcript-Tails. Codex kommt erst ins Spiel, wenn du auf einer Karte „Analysieren" oder „Folgeprompt" drückst.

Slices1 (Board) + 2 (Reports)
LLM-KostenNull — reine Metadaten
QuellenRuntimeWatcher · HookBridge · Snapshot
00 ✱ ÜBERSICHT

Was dieses Paket leistet.

Der Kern-Use-Case: fünf Projekte parallel, drei Agents laufen, du sitzt in einem Call — und weißt trotzdem in einer Sekunde, wo es klemmt.

Input

Was rein geht

  • Runtime-Statuswechsel aus AgentSessionRuntimeWatcher (vnode-Events + Poll-Fallback)
  • Hook-Events für Claude-Sessions aus ClaudeHookBridge (Notification = awaitingInput, ohne Polling)
  • Workspace-Snapshot aus JarvisWorkspaceSnapshotBuilder (Sessions, Projekte, Tabs, Pins, PTY-Status)
  • Transcript-Tails aus den bestehenden Readern — letzte Assistant-Message als Einzeiler
Output

Was rauskommt

  • Board — Sessions gruppiert nach Attention-Kategorie, live sortiert
  • Tier-1-Report-Karten — sofort beim Statuswechsel, mit Ein-Klick-Buttons
  • Badge-Zahl — Sessions in „braucht dich", im Inspector-Header und an der MenuBarExtra
  • macOS-Notifications (Report Mode On, App unfokussiert) via AgentSessionNotifier
Leitplanke

Diese Schicht muss vollständig funktionieren, wenn Codex fehlt. Sie ist der Fallback und gleichzeitig das Herz des Features — wer nie mit Jarvis chattet, bekommt trotzdem den Überblick.

01 ✱ ATTENTION-MODELL

Sortiert nach „braucht mich".

Purer, unit-getesteter Builder: JarvisAttentionModelBuilder nimmt Snapshot + Read-Flags und liefert die Board-Struktur. Keine View-Abhängigkeiten.

1 · awaitingInput
Höchste Priorität. Die Session stellt dir eine Frage und blockiert, bis du antwortest. Quelle: RuntimeWatcher-Ableitung + awaitingInputSessionIDs-Pulse + Claude-Notification-Hook. Zählt in den Badge.
2 · errored
Gestorben, braucht Diagnose. Karte zeigt die letzte Fehlerzeile aus dem Tail. „Analysieren"-Button ist hier am wertvollsten. Zählt in den Badge.
3 · fertig · ungelesen
Ergebnis liegt bereit. working → idle/stopped mit ungelesenem Tier-1-Report. „Erledigt" markiert gelesen und stuft die Session auf Kategorie 5 zurück. Zählt in den Badge.
4 · working
Läuft — nur beobachten. Zeigt Dauer seit Turn-Start und den aktuellen Arbeitsschritt aus dem Tail (z. B. „schreibt Tests"). Zählt nicht in den Badge.
5 · idle / Rest
Ruhig. Gedimmt dargestellt (55 % Opacity), nach lastActivityAt sortiert. Archivierte Sessions erscheinen gar nicht.
Badge-Semantik

Der Badge zählt Sessions in Kategorie 1–3 — nicht Reports. Ein Zähler, der mit jedem Report hochtickt, wird zur Mail-Inbox mit 4.000 Ungelesenen: man stumpft ab. „Erledigt" senkt den Zähler sofort.

States

Board-Zustände

  • Gruppen-Header: „BRAUCHT DICH" amber, „LÄUFT" neutral, „RUHIG" gedimmt; Zähler pro Gruppe
  • Karten-Rahmen färbt nach Dringlichkeit: amber (awaitingInput), rot (errored), neutral sonst
  • Ruhig-Zeilen mit 55 % Opacity — sichtbar, aber ohne Gewicht
  • Leerzustand: positive Bestätigung + verbleibende „läuft"-Sessions, kein leeres Grau
Interactions

Klick-Verhalten

  • Klick auf Zeile = Session im Fenster selektieren/öffnen (bestehende Open-Tab-Pfade, Host-Callback)
  • „Öffnen" identisch, aber explizit — für die Karten mit Buttons
  • „Erledigt" setzt isRead, Session fällt aus Kategorie 3, Badge sinkt sofort
  • „Analysieren" = Tier-2-Brain-Turn für genau diese Session (Abschnitt 03)
  • „Folgeprompt" = Brain-Turn, Ergebnis landet als Draft unter Aktionen

Tier-1-Regeln

Was der Builder kann — und was nicht

  • Summary = letzte Assistant-Message aus dem Tail, hart gekürzt (~240 Zeichen)
  • Dauer aus lastTurnAt → Statuswechsel; Trigger + Projekt aus dem Snapshot
  • Kein Urteil, keine Interpretation — das ist ehrlicher als ein schlechter LLM-Einzeiler
  • Fehlt der Tail (missingTranscript): nur Metadaten-Zeile, Karte bleibt nutzbar
Tier-2-Regeln

Anreicherung on demand

  • Genau ein Codex-Call, volles Digest-Budget (12k) nur für diese Session
  • Ergebnis überschreibt die Karte nicht — es ergänzt Outcome/Blocker/NextStep
  • „Folgeprompt" erzeugt zusätzlich einen JarvisPromptDraft, verlinkt via suggestedPromptDraftID
  • Während des Turns: Button zeigt Spinner, Rest des Boards bleibt bedienbar (serielle Queue)

Regeln

Wann eine Notification rausgeht

  • Nur bei Report Mode On und nur für Kategorie 1–3 (awaitingInput, errored, fertig)
  • Nur wenn kein Agent-Chats-Fenster fokussiert ist — sonst reicht der Badge
  • Dedupe identisch zu Tier-1-Reports: max. eine pro Session + Transcript-Version
  • Bestehenden AgentSessionNotifier-Pfad wiederverwenden, keine zweite Notification-Pipeline
Interactions

Klick-Ziele

  • Klick auf Notification → Agent-Chats-Fenster aktivieren + Zielsession selektieren
  • MenuBar-Badge hängt am bestehenden ✦-Item; Klick öffnet das Fenster mit Übersicht-Segment
  • Badge-Zahl ist dieselbe Quelle wie im Inspector-Header — ein Attention-Modell, zwei Anzeigen
03 ✱ TIER-1-PIPELINE

Vom Statuswechsel zur Karte.

Trigger, Dedupe und die Grenze zwischen kostenlos und Codex.

Trigger
working → idle, working → stopped, working → awaitingInput, working → errored. Für Claude-Sessions zusätzlich event-driven über die Notification-Hooks der ClaudeHookBridge.
Dedupe
Max. ein Report pro Session pro Transcript-Version (Version = Digest-Cache-Key aus path + size + mtime). Reine UI-Selektionswechsel erzeugen nie einen Report.
Latenz
Karte erscheint < 1 s nach dem Statuswechsel — der Builder liest nur den gecachten Tail. Kein Codex-Prozess, keine Queue-Wartezeit.
Tier-2-Grenze
Codex läuft nur nach explizitem Klick („Analysieren", „Folgeprompt") oder manuellem Briefing über alle ungelesenen Reports. Auto-Batch mit Debounce ist bewusst v2.
Prune
Inbox behält die letzten 200 Reports; ältere fliegen beim Persistieren raus. Gelesene Reports bleiben scrollbar (Feed-Historie), zählen aber nicht in den Badge.

05 ✱ EDGE-CASES

Was schief gehen kann.

Transcript fehlt
Digest liefert missingTranscript → Karte ohne Einzeiler, nur Metadaten (Trigger, Projekt, Dauer). „Analysieren" bleibt aktiv, arbeitet dann nur mit Snapshot-Kontext.
Keine externalSessionID
Frisch gestartete Session, Hook-Bind noch nicht durch → missingExternalSessionID. Board zeigt die Session mit Snapshot-Daten; Tail folgt, sobald die ID gebunden ist.
Statuswechsel-Sturm
Session flappt working ↔ idle (z. B. Tool-Pausen): Dedupe pro Transcript-Version fängt das — gleiche Version, kein neuer Report. Erst neuer Inhalt erzeugt neue Karten.
AgentView-Session
unsupportedAgentView — kein Subsession-Digest in v1. Zeile erscheint im Board mit Status, aber ohne Einzeiler; „Analysieren" ist deaktiviert mit Tooltip.
Mehrere Fenster
Alle Fenster rendern dasselbe globale Attention-Modell. „Erledigt" in Fenster A senkt den Badge auch in Fenster B und an der MenuBar — eine Quelle, drei Anzeigen.
200-Report-Grenze
Prune entfernt älteste zuerst, aber nie einen ungelesenen Report einer noch existierenden Session — sonst verschwindet ein „braucht dich" kommentarlos.
