01 ✱ DIGEST

Routing und Cache.

Pro Session ein Digest — aber der Weg zum Transcript hängt am Provider und Kind. AgentChatTailExtractor ist die Vorlage.

Claude · .chat
ClaudeTranscriptReader via externalSessionID + cwd → ~/.claude/projects/<encoded-cwd>/<id>.jsonl
Codex · .chat
CodexTranscriptReader via externalSessionID → ~/.codex/sessions/YYYY/MM/DD/*.jsonl
Claude · .backgroundChat
SupervisorJobReader.readSingle(shortID) → linkScanPath → ClaudeTranscriptReader
.agentView
unsupportedAgentView — kein opportunistischer Subsession-Digest in v1. UI kommuniziert das klar (deaktivierter Analysieren-Button).
Cache
In-Memory-LRU, ~50 Einträge. Key = sessionID + transcriptPath + fileSize + mtime. Optionaler Warm-Start-Sidecar jarvis-digest-cache.json — nie Teil von jarvis-state.json.
Budget
Voll (~12k): @-Mentions, Selektion, frische Statuswechsel. Metadaten (~200): Rest. Gesamt ~60k, Kürzung in Attention-Reihenfolge, Kürzungen im Prompt vermerkt.
02 ✱ BRAIN

Der Codex-Aufruf, exakt.

Genau ein Prozess pro Turn, seriell. CodexInvocation aus PostProcessingService ist die produktiv erprobte Vorlage — Flags vor Slice 1 gegen die installierte Version verifizieren.

# JarvisCodexClient — Aufbau des Aufrufs codex exec \ --sandbox read-only # Jarvis liest nur — Mutationen macht die App --skip-git-repo-check \ --output-last-message <temp-output-path> \ -m <model> # Default aus bestehenden Codex-Settings -c model_reasoning_effort=<effort> \ - # Prompt via STDIN — nie als argv (ARG_MAX, # Prozessliste); AgentHeadlessCLI bekommt dafür # eine stdin-Pipe. Alternative: Temp-Datei. # Environment: LoginShellEnvironment.shared.processEnvironment() # Binary: AgentCommandBuilder.commandPath("codex") # Timeout: 45 s Chat · 90 s Cleanup/Briefing → danach Kill + Fehler-State # Queue: seriell — nie mehr als ein Codex-Prozess gleichzeitig
Prompt-Aufbau

Was in den Prompt geht (in dieser Reihenfolge)

  • Systemprompt: Rolle, Mutationsverbot, Antwortformat, Report-/Draft-Qualitätsregeln
  • Snapshot kompakt: eine Zeile pro Session (Titel, Projekt, Status, Dauer, Kind)
  • Digests der Budget-Sessions (mit Kürzungsvermerken)
  • Chat-Auszug: letzte N Messages
  • Action-Schemas: nur die 5 mutierenden Kinds mit Feldbeschreibung
  • Auftrag: der User-Turn bzw. Cleanup/Analyse/Briefing-Direktive
Sofort-Pfad

Wann Codex gar nicht startet

  • Quick-Reply-Chips („Wer wartet?", „Was läuft?") → Antwort direkt aus dem Attention-Modell
  • Codex fehlt → Chat deaktiviert mit Setup-Hinweis, Rest der App unberührt
  • Mention nicht auflösbar → Turn startet nicht, Picker fragt nach
03 ✱ JSON-VERTRAG

Strict rein, tolerant gelesen.

Codex muss genau ein JSON-Objekt liefern — kein toolCalls-Feld mehr. Der Parser ist streng in der Struktur, aber tolerant in der Verpackung.

// Antwortformat (JarvisBrainResponse) { "assistantMessage": "Kurzantwort an den Nutzer.", "reports": [ /* Tier-2-Anreicherungen — Draft-Typ ohne IDs */ ], "actions": [ /* Mutations-Vorschläge — Draft-Typ ohne IDs */ ], "promptDrafts": [ /* Folgeprompts — Draft-Typ ohne IDs */ ] }
Parser-Regeln

Reihenfolge der Verarbeitung

  • Fence-Stripping: ```-Fences und Prosa vor/nach dem Objekt entfernen, dann strikt decodieren
  • IDs, createdAt, status vergibt die App — modellgenerierte UUIDs wären Kollisions- und Dedupe-Gift
  • Session-Referenzen (Titel/Kontext) gegen den Snapshot mappen; unauflösbar = verwerfen + loggen
  • Unbekannte Action-Kinds ignorieren und loggen
  • Kaputt trotz Stripping → Raw-Output als failed Chat-Event, keine Actions
Validierung

Inhaltliche Ablehnungen

  • Archive abgelehnt bei working oder laufendem PTY
  • Send abgelehnt ohne laufenden Terminal-Controller
  • Rename bei manuellem Titel nur mit explizitem Cleanup-Flag der Action
  • Behauptete Mutation ohne Action-Draft → Systemhinweis an die Message („keine Aktion erzeugt")
04 ✱ PERSISTENZ

Zwei Dateien, klare Diät.

jarvis-state.json
~/Library/Application Support/WhisperM8/ — Chat (200), Reports (200), Actions (100), Drafts (100), reportModeEnabled. Atomic writes analog saveUIState, debounced, Flush bei willTerminate. Ein globaler Store (Registry-Pattern), Tests mit Temp-URL.
jarvis-digest-cache.json
Optionaler Warm-Start-Sidecar des In-Memory-LRU. Bewusst getrennt: 50 × 12k Zeichen gehören nicht in eine Datei, die bei jeder Chat-Message debounced neu geschrieben wird.
Kein Touch
AgentSessions.json, agent-ui-state.json, alles unter ~/.claude / ~/.codex — Jarvis liest über die bestehenden Facades und schreibt Workspace-Daten ausschließlich über AgentSessionStore.
Prune-Sonderregel
Nie einen ungelesenen Report einer existierenden Session prunen — ein „braucht dich" darf nicht kommentarlos verschwinden.
05 ✱ GOTCHAS

Die Stellen, an denen es beißt.

Aus dem Architektur-Review — jede Zeile hier hat einen konkreten Bug verhindert, bevor er geschrieben wurde.

Store-Lock
Mutation-Closures im Workspace-Store laufen unter dem prozessweiten Lock — keine Subprozesse, kein I/O, keine Git-Lookups darin. Apply-Validierung komplett vor dem mutate-Aufruf hoisten.
MainActor-Grenze
AgentTerminalRegistry und UI-Modelle sind MainActor-gebunden. Der Snapshot-Builder sammelt auf dem MainActor und übergibt eine Sendable-Kopie; Digest + Codex laufen off-main.
stdin statt argv
60k-Prompts als argv-Argument riskieren ARG_MAX und stehen in der Prozessliste. AgentHeadlessCLI bekommt eine stdin-Pipe — die einzige Erweiterung an bestehendem Code neben sendPrompt.
LoginShellEnvironment
GUI-Apps haben minimalen launchd-PATH. Nie codex mit der rohen ProcessInfo-Environment spawnen — immer LoginShellEnvironment + AgentCommandBuilder.commandPath.
Bracketed Paste
Mehrzeilige Prompts roh in die PTY = jede Newline submitted. ESC[200~ … ESC[201~ + Return mit 50–100 ms Delay danach.
Persistierter Thinking-State
App-Neustart mid-Turn: offene Turns beim Start als cancelled markieren. Ein aus JSON wiederauferstandener THINKING-Header wäre ein ewiger Spinner.
