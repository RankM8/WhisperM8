01 ✱ GO/NO-GO

Der Check nach Slice 1.

Bevor Slices 2–4 gebaut werden, wird mit dem realen Workspace bewertet. Scheitert der Check, ist der Schaden ein Slice — nicht vier.

GO-Kriterien

Was Slice 1 zeigen muss

  • Antwortqualität: Snapshot-Fragen („was liegt an?") werden konkret und korrekt beantwortet — keine Halluzinationen über Sessions
  • Latenz: typischer Chat-Turn unter ~20 s Ende-zu-Ende; 45 s werden nie ohne Ergebnis gerissen
  • Format-Treue: > 95 % der Antworten parsen nach Fence-Stripping strikt
  • Board-Wert: das LLM-freie Board fühlt sich schon allein nützlich an
NO-GO-Pfade

Wenn es scheitert

  • Latenz zu hoch → kleineres Modell / geringerer Effort testen, bevor die Engine in Frage steht
  • Format bricht ständig → Prompt härten; notfalls Antwort-Schema vereinfachen (nur assistantMessage + drafts)
  • Qualität schwach → Engine-Frage neu stellen (claude headless als Alternative) — Architektur bleibt, nur der Client tauscht
  • Das Board bleibt in jedem Fall — es hängt nicht an Codex
02 ✱ TESTPLAN

Was grün sein muss.

Repo-Konvention: DI via Closures und kleine Protokolle, Temp-URLs für Stores, pure Builder ohne View-Abhängigkeiten. Die Drag/Drop-artigen Interaktionen bleiben manuelle QA.

Store
JarvisStateStore load/save/prune (200/200/100/100); Prune-Sonderregel: nie ungelesene Reports existierender Sessions.
Snapshot
Enthält alle Workspace-Sessions + alle Fenster-Tabs, aber keine historischen Disk-Sessions; Sendable-Kopie korrekt.
Attention
Sortierung awaitingInput > errored > fertig-ungelesen > working > idle; Badge zählt nur Kategorie 1–3; „Erledigt" stuft zurück.
Digest
Cache invalidiert bei mtime/size-Änderung; LRU-Bound greift; Background liest linkScanPath; AgentView liefert unsupportedAgentView.
Tier-1
Report pro Trigger; Dedupe pro Session + Transcript-Version; kein Report bei reiner Selektionsänderung; Summary-Kürzung.
Parser
Valides JSON akzeptiert; Fences gestrippt; unknown Kinds verworfen + geloggt; IDs app-seitig; nicht auflösbare Session-Referenzen verworfen.
Prompt Builder
Budget-Kürzung in Attention-Reihenfolge mit Vermerk; @-Mention hebt Session ins volle Budget; Projekt-Mention zieht aktive Sessions nach.
Mentions
JarvisMentionResolver: Fuzzy über Titel + Projekt; Mehrdeutigkeit wird gemeldet, nicht geraten; tote Referenz schlägt fehl.
Actions
rename/group/archive validieren Zielsession und schreiben korrekt über AgentSessionStore; Drift → failed; Undo stellt before her; Archive abgelehnt bei working/PTY; Rename-Schutz bei manuellem Titel.
Send
Nur bei laufendem Controller erlaubt; mehrzeiliger Prompt = genau ein Bracketed-Paste-Block + optionales Return nach Delay.
UI / Integration
Jedes Fenster rendert denselben globalen State; simulierter Statuswechsel → sofortiger Feed-Eintrag ohne Codex-Prozess; Apply/Skip/Undo aktualisiert Status; Codex-fehlt-Fallback lässt Board voll funktionieren.
Manual Acceptance — die fünf Szenen

① 5+ parallele Sessions, Board sortiert live. ② Claude-Session stoppen → Tier-1 sofort, „Analysieren" liefert Brauchbares, kein ungefragter Codex-Prozess. ③ Report → Folgeprompt → Send: ein Block, ein Submit. ④ „Aufräumen" → sinnvolle Vorschläge, nichts ändert sich ohne Apply, Undo funktioniert. ⑤ awaitingInput bei App im Hintergrund → Notification → Klick landet in der Session.

03 ✱ FAILURE MODES

Degradieren, nicht sterben.

Codex fehlt / abgemeldet
Board, Tier-1-Reports, Actions-Historie: voll funktionsfähig. Chat deaktiviert mit Setup-Hinweis, Header zeigt CODEX ERROR.
Kaputtes JSON
Auch nach Fence-Stripping unparsebar → Raw-Output als failed Chat-Event einsehbar, keine Actions erzeugt.
Transcript fehlt
missingTranscript — Tier-1 nutzt Metadaten, Karte bleibt nutzbar.
Keine externalSessionID
missingExternalSessionID — Board zeigt Snapshot-Daten, Tail folgt nach Hook-Bind.
Background-Short-ID fehlt
missingTranscript bzw. missingExternalSessionID je nach Pfad — nie ein Crash im Digest-Routing.
PTY nicht running
Send deaktiviert; Copy und „Tab öffnen & einfügen" übernehmen.
Workspace-Drift
Apply-Revalidierung schlägt an → Action failed mit konkreter Meldung, nie stilles Überschreiben.
Statuswechsel mid-Turn
Tier-1 entsteht sofort, unabhängig vom laufenden Brain-Turn; die serielle Queue arbeitet danach weiter.
App-Neustart mid-Turn
Offene Turns beim Start als cancelled markiert — kein persistierter Ewig-Spinner.
Mehrere Fenster
Ein globaler State, fensterlokale Sichtbarkeit. Doppel-Apply wird zum No-op mit Hinweis.
04 ✱ V2-KANDIDATEN

Bewusst verschoben.

Alles hier ist entschieden „nicht jetzt" — mit dem v1-Fundament so geschnitten, dass nichts davon einen Umbau braucht.

MCP-Server
Read-only-Tools als echter Tool-Loop, exportiert über einen lokalen MCP-Server — die Action-Schemas sind ab v1 MCP-kompatibel benannt. Dann können auch externe Agents die Jarvis-Sicht nutzen.
Auto-Batch-Briefing
Debounced Tier-2-Zusammenfassung ungelesener Reports („Morning Briefing"). Erst beobachten, wie oft der manuelle Briefing-Chip gedrückt wird.
Auto-Apply reversibel
Opt-in Auto-Apply für Rename/Group — der Undo-Pfad existiert ab v1, das Vertrauen muss er sich erst verdienen.
Auto-Send
Pro Session als Opt-in. Höchste Autonomie-Stufe, kommt zuletzt.
AgentView-Subsessions
Erkennung der selektierten Subsession in claude agents — heute unsupportedAgentView.
Disk-Inventur
Historische ~/.claude / ~/.codex Sessions global inventarisieren. V1 bleibt bei WhisperM8-bekannten Sessions.
Eigenes Fenster / Command-Center
Jarvis als separates Fenster oder MenuBar-Command-Center — erst, wenn der Inspector-Platz nachweislich zu eng wird.
Browser-Integration
Claude-Web-Tabs via Accessibility/Browser-Automation beobachten — außerhalb des v1-Vertrauensmodells.
