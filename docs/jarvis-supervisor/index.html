01 ✱ TRENNUNG

Drei Schichten, klar getrennt.

Die wichtigste Architektur-Entscheidung aus dem Review: Was ohne LLM geht, läuft ohne LLM. Was denkt, ist genau ein Codex-Aufruf. Was verändert, braucht deinen Klick.

SCHICHT 1 ✱ IMMER AN · LLM-FREI

Mission Control

Board, Attention-Sortierung, Tier-1-Reports, Badge, Notifications. Entsteht aus Snapshot + Transcript-Tails. Funktioniert auch, wenn Codex fehlt oder abgemeldet ist.

  • JarvisWorkspaceSnapshotBuilder
  • JarvisAttentionModelBuilder
  • JarvisTier1ReportBuilder
  • Trigger: RuntimeWatcher + ClaudeHookBridge
SCHICHT 2 ✱ ON DEMAND · 1 CALL/TURN

Brain (Codex headless)

Chat, Tier-2-Analysen, Folgeprompt-Entwürfe, Cleanup-Vorschläge. Kontext wird vorab injiziert — kein Tool-Loop, keine Überraschungs-Prozesse, serielle Queue.

  • JarvisCodexClient via codex exec
  • Strict-JSON-Antwort, Fence-tolerant geparst
  • IDs vergibt die App, nie das Modell
  • Budget: 12k Zeichen für erwähnte Sessions, ~200 sonst
SCHICHT 3 ✱ NUR MIT KLICK

Review-Layer

Jede Mutation ist eine Karte: Vorher/Nachher, Begründung, Apply/Skip — und Undo für Rename/Group/Archive. Kein Auto-Send, kein Batch für Archive oder PTY-Send.

  • JarvisActionRegistry (nur mutierend)
  • Apply-Revalidierung + Drift-Erkennung
  • PTY-Send mit Bracketed Paste
  • Pfad: immer über AgentSessionStore
02 ✱ DATENFLUSS

Vom Statuswechsel zum nächsten Prompt.

Der Kern-Loop, den Jarvis dir abnimmt: Session wird fertig → du erfährst es sofort → ein Klick entwirft den Folgeprompt → ein Klick schickt ihn in die laufende PTY.

INPUT ✱ EVENT-DRIVEN

Runtime-Statuswechsel

AgentSessionRuntimeWatcher (vnode + Poll-Fallback) und ClaudeHookBridge melden working → idle / stopped / awaitingInput / errored.

sofort · ohne LLM TIER 1
PROCESS ✱ LLM-FREI

Tier-1-Report + Board-Update

JarvisTier1ReportBuilder baut aus Snapshot + Transcript-Tail einen Metadaten-Report. Dedupe pro Session + Transcript-Version. Board sortiert neu, Badge zählt „braucht mich".

Nutzer klickt „Analysieren" / „Folgeprompt" / fragt im Chat TIER 2 · 1 CODEX-CALL
PROCESS ✱ CODEX HEADLESS

Brain-Turn

Prompt Builder injiziert Snapshot + selektive Digests (@-Mentions, Selektion, frische Wechsel) + Chat-Auszug + Action-Schemas. codex exec --sandbox read-only, Prompt via stdin, Antwort als Strict JSON.

Parser vergibt IDs, validiert Actions
OUTPUT ✱ REVIEW-FIRST

Chat-Antwort · Action-Karten · Prompt-Drafts

Antwort in den Chat, Mutationen als Karten (Apply/Skip/Undo), Drafts mit Copy / Tab öffnen / Senden — Send nur bei laufendem PTY, als Bracketed-Paste-Block.

QUER DAZU ✱ IMMER

Ein Brain, seriell.

Brain-Turns laufen über eine serielle Queue — nie mehr als ein Codex-Prozess. Tier-1 entsteht davon unabhängig und sofort.

Digest-Cache: in-memory LRU, Key aus path + size + mtime. Nicht in jarvis-state.json — die Datei bleibt klein.

Fällt Codex aus, bleibt Schicht 1 vollständig nutzbar: Board, Reports, Badge, Historie.

05 ✱ OFFENE PUNKTE

Was noch zu klären ist.

01

Codex-Flags gegen installierte Version verifizieren

Insbesondere --ephemeral. CodexInvocation.arguments(...) in PostProcessingService ist die erprobte Vorlage — vor Slice 1 einmal real gegen codex exec testen.

VOR SLICE 110-Minuten-Spike
02

Modell/Effort-Defaults für den Brain

Plan sagt: bestehende Codex-Settings wiederverwenden, keine neue Settings-Flut. Offen ist, ob Chat-Turns und Cleanup-Turns unterschiedliche Efforts brauchen (schnell vs. gründlich).

SLICE 1Am Go/No-Go entscheiden
03

Digest-Budgets kalibrieren

12k / 200 Zeichen und 60k Gesamtbudget sind Startwerte. Nach Slice 2 mit realen Transcripts prüfen: reicht der Tail für brauchbare Tier-2-Analysen?

SLICE 2Empirisch tunen
04

Notification-Verhalten bei fokussierter App

Tier-1-Notifications nur, wenn kein Agent-Chats-Fenster fokussiert ist — oder auch bei fokussiertem Fenster, wenn das Panel zu ist? Default: nur unfokussiert, Rest übernimmt der Badge.

SLICE 4UX-Detail
05

Auto-Batch-Briefing

Debounced Tier-2-Zusammenfassung aller ungelesenen Reports („Morning Briefing"). Bewusst v2 — erst sehen, wie oft man den manuellen Briefing-Button wirklich drückt.

V2Entschieden verschoben
06

Read-only-Tools als MCP-Server

Der v1-Schnitt (Kontext vorab injizieren) ist entschieden. Die Action-Schemas werden aber MCP-kompatibel benannt, damit v2 sie 1:1 exportieren kann.
